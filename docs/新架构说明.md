# Echo Agentic Engine 新架构说明

## Redis 表结构

### 1. 引擎信息表 (engine:*)

**键格式**: `engine:{engine_url}`

**字段**:
- `engineUrl`: 引擎完整URL (如 http://192.168.1.100:8000/trans)
- `maxSandboxes`: 支持的最大沙箱数量
- `usedSandboxes`: 当前已使用的沙箱数量
- `timestamp`: 最后更新时间戳

**示例**:
```
Key: engine:http://192.168.1.100:8000/trans
Fields:
  engineUrl: "http://192.168.1.100:8000/trans"
  maxSandboxes: "100"
  usedSandboxes: "15"
  timestamp: "2026-03-20T10:30:00Z"
```

### 2. 沙箱信息表 (sandbox:*)

**键格式**: `sandbox:{domainID}:{sandboxID}`

**字段**:
- `domainID`: 租户标识
- `sandboxID`: 沙箱标识
- `engineUrl`: 所属引擎完整URL
- `timestamp`: 最后更新时间戳
- `lastRequestTime`: 最近一次请求处理完成时间

**示例**:
```
Key: sandbox:tenant-001:sandbox-001
Fields:
  domainID: "tenant-001"
  sandboxID: "sandbox-001"
  engineUrl: "http://192.168.1.100:8000/trans"
  timestamp: "2026-03-20T10:30:00Z"
  lastRequestTime: "2026-03-20T10:25:30Z"
```

## ConfigSyncTask 流程

**执行间隔**: 5秒

**职责**:
1. 向Redis写入引擎信息表
2. 检测并清理不健康沙箱
3. 同步沙箱信息表

**流程**:
```
每5秒执行:
├─ 1. 获取引擎信息
│   ├─ 引擎URL (http://ip:port/trans)
│   ├─ 最大沙箱数 (从config.json读取)
│   ├─ 当前沙箱数 (从内存获取)
│   └─ 当前时间戳
│
├─ 2. 写入Redis引擎信息表
│   └─ engine:{engine_url}
│
├─ 3. 检测并清理不健康沙箱
│   ├─ 并发检查所有沙箱健康状态
│   ├─ 对不健康沙箱执行:
│   │   ├─ 销毁沙箱进程
│   │   └─ 从Redis删除沙箱信息
│   └─ 记录日志
│
└─ 4. 同步沙箱信息表
    ├─ 从Redis读取本引擎的所有沙箱记录
    ├─ 获取内存中的鲜活沙箱列表
    ├─ 多退: 删除Redis中已不存在的沙箱记录
    └─ 少补: 添加新沙箱记录到Redis
```

## 管理接口

### 3.2.2 列出所有信息

**接口**: `POST /admin/sandbox/list`

**请求体**: `{}`

**响应**:
```json
{
  "status": "success",
  "data": {
    "engines": [
      {
        "engineUrl": "http://192.168.1.100:8000/trans",
        "maxSandboxes": 100,
        "usedSandboxes": 15,
        "timestamp": "2026-03-20T10:30:00Z"
      }
    ],
    "sandboxes": [
      {
        "domainID": "tenant-001",
        "sandboxID": "sandbox-001",
        "engineUrl": "http://192.168.1.100:8000/trans",
        "timestamp": "2026-03-20T10:30:00Z",
        "lastRequestTime": "2026-03-20T10:25:30Z"
      }
    ],
    "summary": {
      "totalEngines": 1,
      "totalSandboxes": 15
    }
  }
}
```

### 3.2.3 清理空闲沙箱

**接口**: `POST /admin/sandbox/get`

**请求体**:
```json
{
  "idleMinutes": 5
}
```

**功能**: 清理空闲超过指定时长(分钟)的沙箱

**响应**:
```json
{
  "status": "success",
  "data": {
    "idleMinutes": 5,
    "totalScanned": 15,
    "destroyed": 3,
    "failed": 0
  }
}
```

**说明**:
- 只清理本引擎负责的沙箱
- echo-agent可以调用此接口清理所有空闲沙箱
- 通过比较lastRequestTime和当前时间判断空闲时长

### 3.2.4 创建沙箱

**接口**: `POST /admin/sandbox/create`

**请求体**:
```json
{
  "domainID": "tenant-001",
  "sandboxID": "sandbox-001"
}
```

**流程**:
1. 创建沙箱进程
2. 写入Redis沙箱信息表
3. 返回沙箱信息

### 3.2.5 销毁沙箱

**接口**: `POST /admin/sandbox/destroy`

**请求体**:
```json
{
  "domainID": "tenant-001",
  "sandboxID": "sandbox-001"
}
```

**流程**:
1. 销毁沙箱进程
2. 从Redis删除沙箱信息
3. 返回销毁结果

## 已删除的流程

### 删除原因

以下两个后台任务已被ConfigSyncTask替代:

1. **流程 3.2.7: HealthCheckTask 调用链**
   - 功能: 定期检查所有沙箱健康状态并重建不健康沙箱
   - 替代方案: ConfigSyncTask每5秒检查健康状态,发现不健康沙箱直接杀死

2. **流程 3.2.8: IdleCleanupTask 调用链**
   - 功能: 定期清理空闲超过阈值的沙箱
   - 替代方案: echo-agent通过调用3.2.3接口主动清理空闲沙箱

### 架构优势

**原架构问题**:
- Engine自行管理空闲沙箱清理,无法与echo-agent协同
- 健康检查和空闲清理分散在两个任务中,职责不清

**新架构优势**:
- Engine专注于沙箱管理和健康检查
- echo-agent负责全局调度和空闲清理策略
- 通过Redis表共享状态,支持多Engine实例
- 定时同步机制确保数据一致性

## 配置文件说明

**config.json 新结构**:
```json
{
  "engine": {
    "host": "0.0.0.0",
    "port": 8000,
    "maxSandboxes": 100,
    "logLevel": "INFO"
  },
  "sandbox": {
    "portRange": {
      "min": 30001,
      "max": 40000
    },
    "idleTimeout": 7200,
    "openCodeCommand": "/usr/local/bin/opencode"
  },
  "tasks": {
    "configSyncInterval": 5,
    "healthCheckTimeout": 5,
    "maxRetries": 3
  }
}
```

**配置变更说明**:
- `tasks.configSyncInterval`: 从300秒改为5秒
- `tasks.healthCheckInterval`: 已删除
- `tasks.idleCheckInterval`: 已删除
- `engine.maxSandboxes`: 从50改为100
- `sandbox.idleTimeout`: 从3600秒改为7200秒
