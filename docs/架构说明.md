# Echo Agentic Engine 架构说明

## 核心概念

### 沙箱隔离架构

Echo Agentic Engine 使用 Bubblewrap 实现轻量级沙箱隔离，每个 OpenCode Serve 实例运行在独立的沙箱中。

### 目录结构

每个沙箱在宿主机上的目录结构：

```
{nas-root}/{domainID}/{sandboxID}/
├── config/      # XDG配置目录（挂载到沙箱/config）
├── data/        # XDG数据目录（挂载到沙箱/data）
├── workspace/   # 工作目录（挂载到沙箱/workspace）
└── tmp/         # 临时文件目录（挂载到沙箱/tmp）
```

**说明：**
- `nas-root`: 配置文件中定义的 NAS 根路径，默认为 `/data/users`
- `domainID`: 租户标识，用于多租户隔离
- `sandboxID`: 沙箱标识，用于同一租户下的不同沙箱实例

### Bubblewrap 沙箱挂载

每个沙箱启动时的挂载结构：

```bash
bwrap \
  --ro-bind /usr /usr \                      # 只读挂载系统目录
  --ro-bind /etc/resolv.conf /etc/resolv.conf \
  --ro-bind /etc/ssl /etc/ssl \
  --ro-bind /lib /lib \
  --ro-bind /lib64 /lib64 \
  --ro-bind /bin /bin \
  --ro-bind /sbin /sbin \
  --ro-bind /proc /proc \
  --dev /dev \                               # 设备文件系统
  --bind {sandbox-root}/data /data \         # 读写挂载数据目录
  --bind {sandbox-root}/config /config \     # 读写挂载配置目录
  --bind {sandbox-root}/workspace /workspace \ # 读写挂载工作目录
  --bind {sandbox-root}/tmp /tmp \           # 读写挂载临时目录
  --unshare-user \                            # 隔离用户命名空间
  --unshare-pid \                             # 隔离PID命名空间
  --die-with-parent \                         # 父进程退出时终止
  --new-session \                              # 创建新会话
  --chdir /workspace \                         # 工作目录
  --setenv PATH "/usr/local/bin:/usr/bin:/bin" \
  --setenv HOME "/tmp" \
  --setenv XDG_DATA_HOME "/data" \
  --setenv XDG_CONFIG_HOME "/config" \
  --setenv XDG_STATE_HOME "/data/state" \
  --setenv XDG_CACHE_HOME "/data/cache" \
  -- \
  opencode serve --hostname=127.0.0.1 --port {port}
```

### OpenCode 多项目支持

**重要说明：**
- Engine 只负责创建沙箱根目录和 `workspace` 目录
- `workspace` 目录作为 OpenCode Serve 的工作目录
- OpenCode Serve 本身支持多项目管理，项目管理通过 OpenCode API 实现
- Engine 不关心 workspace 内部的项目结构，由 OpenCode Serve 自行管理

**OpenCode 配置：**
- 配置文件位置：`~/.config/opencode/opencode.json`
- Engine 通过 bwrap 的 `--setenv XDG_CONFIG_HOME /config` 设置配置目录
- 每个沙箱可以有自己的 OpenCode 配置

### Engine IP 信息

每个沙箱的元数据包含 Engine IP 信息，用于多 Engine 实例场景：

```json
{
  "domainID": "test-domain",
  "sandboxID": "test-sandbox",
  "pid": 12345,
  "port": 30001,
  "nasPath": "/data/users/test-domain/test-sandbox",
  "password": "uuid-password",
  "engineHost": "192.168.1.100",
  "status": "running",
  "createdAt": "2024-01-01T00:00:00Z",
  "lastActiveAt": "2024-01-01T00:00:00Z"
}
```

**用途：**
- 上游 echo-agent 根据 Redis 中的沙箱信息获取 Engine IP
- 支持多 Engine 实例的负载均衡和故障恢复

## 请求流程

### 代理请求流程

```
Client -> Engine (代理) -> Sandbox (OpenCode Serve) -> Response
```

**步骤：**
1. Client 发送请求到 Engine，携带 Headers：
   - `X-Domain-ID`: 租户标识
   - `X-Sandbox-ID`: 沙箱标识
2. Engine 验证 Headers
3. Engine 从 Redis 或内存获取/创建沙箱实例
4. Engine 将请求代理到沙箱的 OpenCode Serve
5. OpenCode Serve 处理请求并返回响应
6. Engine 返回响应给 Client

### 流式响应支持

Engine 支持 SSE (Server-Sent Events) 流式响应：
- 自动检测流式请求（路径包含 `/event` 或 `Accept: text/event-stream`）
- 使用 `httpx.AsyncClient.stream()` 进行流式传输
- 返回 `StreamingResponse` 而非普通 `Response`

## 配置说明

### Engine 配置 (config.json)

```json
{
  "redis": {
    "host": "127.0.0.1",
    "port": 6379,
    "password": "",
    "db": 0
  },
  "nas": {
    "rootPath": "/data/users"
  },
  "engine": {
    "host": "192.168.1.100",
    "port": 8000,
    "maxSandboxes": 50,
    "logLevel": "INFO"
  },
  "sandbox": {
    "portRange": {
      "min": 30001,
      "max": 40000
    },
    "idleTimeout": 3600,
    "openCodeCommand": "/usr/local/bin/opencode"
  },
  "tasks": {
    "healthCheckInterval": 30,
    "idleCheckInterval": 60,
    "configSyncInterval": 300,
    "healthCheckTimeout": 5,
    "maxRetries": 3
  }
}
```

**关键配置项：**
- `nas.rootPath`: 沙箱根目录，所有沙箱数据存储在此路径下
- `engine.host`: Engine 对外暴露的 IP 地址，用于多实例场景
- `sandbox.portRange`: 沙箱端口范围，避免端口冲突

## 健康检查机制

Engine 定期检查所有沙箱的健康状态：

1. **健康检查任务**（默认30秒间隔）
   - 发送 `GET /global/health` 到每个沙箱
   - 失败则标记为不健康

2. **自动重建机制**
   - 发现不健康的沙箱自动重建
   - 保留原沙箱的 domainID 和 sandboxID
   - 分配新端口，重新启动进程

3. **空闲清理任务**（默认60秒间隔）
   - 检查沙箱空闲时间
   - 超过 `idleTimeout` 的沙箱自动销毁
   - 回收端口资源

## 故障恢复

### 单点故障

- Engine 进程崩溃：所有沙箱进程（bwrap）会随父进程退出而终止（`--die-with-parent`）
- Redis 连接失败：Engine 转为独立模式运行，无法跨节点恢复

### 多实例部署

- 多个 Engine 实例共享 Redis
- 每个沙箱元数据包含 `engineHost`
- 上层负载均衡器根据 `engineHost` 路由到正确的 Engine
- 支持跨节点故障恢复

## 安全隔离

### Namespace 隔离

- **User Namespace**: 用户权限隔离
- **PID Namespace**: 进程空间隔离

### 文件系统隔离

- **只读挂载**: 系统目录（/usr, /lib, /bin等）
- **读写挂载**: 沙箱专用目录（/data, /config, /workspace, /tmp）

### 网络隔离

- 每个沙箱监听独立的本地端口（127.0.0.1:{port}）
- Engine 通过本地端口代理请求

## 性能优化

### 端口分配

- 端口范围可配置（默认30001-40000）
- 自动分配，自动回收
- 避免端口冲突

### 内存管理

- 定期清理空闲沙箱
- 监控沙箱内存使用
- 防止资源泄漏

### 并发控制

- 最大沙箱数量限制（默认50）
- 使用异步 I/O 处理请求
- 支持流式响应，减少内存占用
