# Background Tasks (后台任务层)

## 层级定位

Background Tasks Layer 包含 Engine 的后台定时任务,负责沙箱巡检、清理和配置同步等后台工作。

## 核心功能

1. **健康检查巡检**: 定期检查所有沙箱健康状态,自动重建不健康沙箱
2. **空闲清理**: 定期扫描空闲超时沙箱,自动销毁
3. **配置同步**: 定期从 Redis 同步配置,触发更新

## 调用关系

- **被触发**: 由 TaskScheduler 定时触发
- **调用下游**:
  - Core Layer: SandboxManager (批量重建/销毁), ConfigManager (配置更新)
  - Monitoring Layer: HealthChecker, IdleMonitor
  - Infrastructure Layer: RedisClient (读取配置)
- **横向依赖**: ConfigSyncTask 可能触发 IdleCleanupTask 更新配置

## 设计原则

- 并发执行: 多个任务可以并发执行
- 失败重试: 单个任务失败不影响其他任务
- 数据保留: 销毁沙箱时保留 NAS 数据

## 模块设计

### 6.1 HealthCheckTask 模块

**职责**: 健康检查巡检任务,定期检查所有沙箱健康状态并重建不健康沙箱

**参与流程**: 1 个流程 (3.2.7)

**执行间隔**: Config.healthCheckInterval (默认 30 秒)

---

#### 6.1.1 execute

**功能描述**: 执行健康检查巡检任务。获取所有沙箱列表，并发检查每个沙箱的健康状态，对不健康的沙箱执行批量重建操作，确保所有沙箱处于正常运行状态。

**流程**: 3.2.7 - HealthCheckTask 调用链

**入参**: 无

**出参**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| (无返回值) | None | 任务执行完成，无返回值 |

**出参示例**:
```python
None
```

**内部调用流程**:
1. SandboxManager.listAllSandboxes() → 获取所有沙箱列表
2. HealthChecker.checkAllHealth(sandboxes) → 并发检查健康状态
   - 对每个沙箱发送: GET http://127.0.0.1:{port}/global/health
   - 返回: {healthy: [...], unhealthy: [...]}
3. SandboxManager.batchRebuild(unhealthySandboxes) → 批量重建不健康沙箱
4. 记录健康检查日志

**依赖模块**:
- `SandboxManager`: 获取沙箱列表、批量重建沙箱
- `HealthChecker`: 检查沙箱健康状态

### 6.2 IdleCleanupTask 模块

**职责**: 空闲沙箱清理任务,定期清理长时间未活跃的沙箱

**参与流程**: 2 个流程 (3.2.8, 3.2.9)

**执行间隔**: Config.idleCheckInterval (默认 60 秒)

---

#### 6.2.1 execute

**功能描述**: 执行空闲沙箱清理任务。获取所有沙箱列表，扫描空闲超时的沙箱（lastActiveAt 超过 Config.idleTimeout），对超时沙箱执行批量销毁操作，释放系统资源。

**流程**: 3.2.8 - IdleCleanupTask 调用链<br>3.2.9 - ConfigSyncTask 调用链（由 ConfigSyncTask 立即触发）

**入参**: 无

**出参**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| (无返回值) | None | 任务执行完成，无返回值 |

**出参示例**:
```python
None
```

**内部调用流程**:
1. SandboxManager.listAllSandboxes() → 获取所有沙箱列表
2. IdleMonitor.scanIdleSandboxes(sandboxes) → 扫描空闲超时沙箱
   - 遍历所有沙箱
   - 计算空闲时长 = now - lastActiveAt
   - 筛选空闲时长 > Config.idleTimeout
   - 返回超时沙箱列表
3. SandboxManager.batchDestroy(timeoutSandboxes) → 批量销毁超时沙箱
   - 对每个超时沙箱执行 destroySandbox()
   - 终止进程、回收端口、删除元数据
4. 记录空闲清理日志

**依赖模块**:
- `SandboxManager`: 获取沙箱列表、批量销毁沙箱
- `IdleMonitor`: 扫描空闲沙箱

### 6.3 ConfigSyncTask 模块

**职责**: 配置同步任务,定期从 Redis 同步配置并处理配置变更影响

**参与流程**: 1 个流程 (3.2.9)

**执行间隔**: Config.configSyncInterval (默认 300 秒)

---

#### 6.3.1 execute

**功能描述**: 执行配置同步任务。从 Redis 读取最新配置，与本地配置进行比较，若有差异则更新本地内存配置，并处理配置变更带来的影响（如 maxSandboxes 减少需销毁多余沙箱，idleTimeout 减少需立即触发空闲清理任务）。

**流程**: 3.2.9 - ConfigSyncTask 调用链

**入参**: 无

**出参**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| (无返回值) | None | 任务执行完成，无返回值 |

**出参示例**:
```python
None
```

**内部调用流程**:
1. RedisClient.getConfig("engine:config") → 获取 Redis 配置
2. ConfigManager.compareConfig(local, redis) → 比较配置差异
   - 遍历所有配置字段
   - 返回差异字段字典
3. 若差异字段为空 → 任务结束
4. ConfigManager.updateConfig(changes) → 更新本地内存配置
5. 配置变更影响处理:
   - **maxSandboxes 减少**:
     - SandboxManager.count() → 获取当前沙箱数量
     - 计算超出数量 = currentCount - newMaxSandboxes
     - 若超出则销毁空闲时间最长的沙箱
   - **idleTimeout 减少**:
     - IdleCleanupTask.execute() → 立即触发空闲清理任务
6. 记录配置同步日志

**配置变更处理规则**:

| 配置项变更 | 处理逻辑 |
|-----------|---------|
| maxSandboxes 减少 | 销毁空闲时间最长的沙箱，直到沙箱数量不超过新上限 |
| idleTimeout 减少 | 立即触发 IdleCleanupTask.execute()，清理超时沙箱 |
| 其他配置变更 | 仅更新内存配置，无需额外处理 |

**依赖模块**:
- `RedisClient`: 获取 Redis 配置
- `ConfigManager`: 比较和更新配置
- `SandboxManager`: 获取沙箱数量、销毁沙箱
- `IdleCleanupTask`: 立即触发空闲清理
