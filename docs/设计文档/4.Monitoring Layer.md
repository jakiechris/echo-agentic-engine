# Monitoring Layer (监控层)

## 层级定位

Monitoring Layer 负责监控沙箱运行状态,包括健康检查、资源监控、空闲监控和日志读取。

## 核心功能

1. **健康监控**: 检查沙箱健康状态,发送 HTTP 健康检查请求
2. **资源监控**: 读取沙箱进程的 CPU、内存占用
3. **空闲监控**: 扫描空闲超时沙箱,触发清理
4. **日志读取**: 读取沙箱 stdout/stderr 日志文件

## 调用关系

- **被调用**: 由 Core Layer 或 API Layer 调用
- **调用外部**:
  - HTTP 客户端: HealthChecker 发送健康检查请求
  - 系统命令: ResourceMonitor 读取进程资源占用
  - 文件系统: LogReader 读取日志文件
- **不调用上层**: 不依赖 Core Layer, API Layer

## 设计原则

- 只读操作: 仅读取状态,不修改沙箱状态
- 异步非阻塞: 健康检查、资源监控使用异步调用
- 超时控制: 健康检查设置超时 (默认 5 秒)

## 模块设计

### 4.1 HealthChecker 模块

**职责**: 沙箱健康检查,通过 HTTP 请求探测沙箱状态

**参与流程**: 5 个流程 (3.2.1 - 3.2.4, 3.2.7)

#### 接口定义

| 方法名 | 流程 | 入参 | 出参 | 说明 |
|--------|------|------|------|------|
| `checkHealth` | 3.2.1, 3.2.3, 3.2.4 | `sandbox: Sandbox` | `str` | 发送 `GET http://127.0.0.1:{port}/global/health`,超时 5 秒,成功且返回 `{"healthy": true}` 返回 `healthy`,否则返回 `unhealthy` |
| `checkAllHealth` | 3.2.2, 3.2.7 | `sandboxes: List[Sandbox]` | `Dict` | 对每个沙箱并发执行 checkHealth(),返回 `{healthy: [...], unhealthy: [...]}` |

---

### 4.2 ResourceMonitor 模块

**职责**: 资源监控,读取沙箱进程的 CPU、内存占用

**参与流程**: 2 个流程 (3.2.2, 3.2.3)

#### 接口定义

| 方法名 | 流程 | 入参 | 出参 | 说明 |
|--------|------|------|------|------|
| `getResourceUsage` | 3.2.2, 3.2.3 | `pid: int` | `ResourceUsage` | 读取 `/proc/{pid}/stat`,解析 RSS、VSZ 字段,计算 CPU 占用率,返回 `{cpu: float, memory: {rss: int, vsz: int, unit: "MB"}}` |

---

### 4.3 IdleMonitor 模块

**职责**: 空闲超时监控,识别长时间未活跃的沙箱

**参与流程**: 1 个流程 (3.2.8)

#### 接口定义

| 方法名 | 流程 | 入参 | 出参 | 说明 |
|--------|------|------|------|------|
| `scanIdleSandboxes` | 3.2.8 | `sandboxes: List[Sandbox]` | `List[Sandbox]` | 遍历所有沙箱,计算空闲时长 = now - lastActiveAt,筛选空闲时长 > Config.idleTimeout,返回超时沙箱列表 |

---

### 4.4 LogReader 模块

**职责**: 日志读取,读取沙箱内 OpenCode 的日志文件

**参与流程**: 1 个流程 (3.2.3)

#### 接口定义

| 方法名 | 流程 | 入参 | 出参 | 说明 |
|--------|------|------|------|------|
| `readLogs` | 3.2.3 | `nasPath: str, limit: int = 50` | `List[str]` | 读取 `{nasPath}/defaultProject/logs/opencode.log`,返回最近 N 行日志 |
