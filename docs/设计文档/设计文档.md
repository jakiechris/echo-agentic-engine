# Echo-Agentic-Engine 设计文档

## 文档说明

本文档是 Engine 服务设计的核心主干，聚焦于三个部分：
1. **对外接口定义** - Engine 提供的所有 API 接口详细定义
2. **接口调用流程** - 每个接口触发的完整调用链和模块交互
3. **模块划分与职责** - 模块的功能边界、输入输出定义

具体模块的实现细节参见同级目录下的独立文档。

---

## 1. 对外接口定义

Engine 对外提供两类接口：
- **业务代理接口**：接收 Echo-Agent 请求，代理到沙箱内 OpenCode Serve
- **管理接口**：沙箱生命周期管理、状态查询

### 1.1 业务代理接口

#### 1.1.1 请求代理接口

**接口路径**: `ANY /trans/{path}`

**功能说明**: 将请求透传到沙箱内 OpenCode Serve，自动处理沙箱创建、健康检查、故障恢复

**请求规范**:

```http
{METHOD} /trans/{path} HTTP/1.1
Host: {engine_host}:{engine_port}
X-Domain-ID: {domainID}
X-Sandbox-ID: {sandboxID}
Content-Type: application/json
[其他 OpenCode 原生 Headers]

{OpenCode 原生请求体}
```

**路径参数**:
- `{path}`: OpenCode 原生 API 路径（如 `api/chat`, `api/files/read`）

**Headers 必需参数**:
- `X-Domain-ID` (string): 租户标识，用于存储隔离和沙箱管理
- `X-Sandbox-ID` (string): 沙箱标识，用于沙箱定位
- 其他 OpenCode 原生 Headers 完整透传

**请求体**:
- 完整透传 OpenCode 原生请求体（JSON 格式）

**响应规范**:

**成功响应** (HTTP 200):
```json
{
  "status": "success",
  "data": {OpenCode 原生响应}
}
```

**失败响应** (HTTP 4xx/5xx):
```json
{
  "status": "error",
  "error": {
    "code": "ERROR_CODE",
    "message": "错误描述",
    "details": {}
  }
}
```

**错误码定义**:

| HTTP 状态码 | 错误码 | 说明 |
|------------|--------|------|
| 400 | MISSING_DOMAIN_ID | 缺少 X-Domain-ID Header |
| 400 | MISSING_SANDBOX_ID | 缺少 X-Sandbox-ID Header |
| 503 | SANDBOX_UNAVAILABLE | 沙箱不可用（创建失败、健康检查失败） |
| 503 | SANDBOX_TIMEOUT | 请求沙箱超时 |
| 500 | INTERNAL_ERROR | Engine 内部错误 |

**行为说明**:

1. **请求验证**: 提取并验证 `X-Domain-ID`, `X-Sandbox-ID`
2. **沙箱检查**: 检查本地是否存在目标沙箱
3. **沙箱创建**: 若不存在，自动创建沙箱（准备 NAS 目录、分配端口、启动 OpenCode Serve）
4. **健康检查**: 若沙箱存在但健康检查失败，自动重建
5. **请求代理**:
   - 路径转换: `/trans/{path}` → `/{path}`
   - Headers 转换: 移除 `X-Domain-ID`, `X-Sandbox-ID`，保留其他
   - 转发到沙箱内: `http://127.0.0.1:{sandbox_port}/{path}`
6. **状态更新**: 更新沙箱活跃时间和 Redis 状态
7. **响应返回**: 完整返回 OpenCode 原生响应

**并发处理**:
- 使用 FastAPI 异步框架，每个请求独立协程处理
- 沙箱重建期间，请求在队列中排队，自动重试 3 次

### 1.2 管理接口

设计理念：所有管理接口使用 POST 方法，方法名在 URL 路径上区分，参数通过 Body 传递。

#### 1.2.1 列出所有沙箱

**接口路径**: `POST /admin/sandbox/list`

**功能说明**: 列出 Engine 管理的所有本地沙箱，包含健康状态、资源占用等信息

**请求规范**:

```http
POST /admin/sandbox/list HTTP/1.1
Content-Type: application/json

{}
```

**请求体**: 无参数（空 JSON 对象）

**响应规范**:

```json
{
  "status": "success",
  "data": {
    "total": 2,
    "sandboxes": [
      {
        "domainID": "tenant-001",
        "sandboxID": "sb-12345",
        "status": "running",
        "pid": 12345,
        "port": 30001,
        "createdAt": "2026-03-14T10:30:00Z",
        "lastActiveAt": "2026-03-14T12:45:30Z",
        "idleSeconds": 120,
        "memoryUsage": {
          "rss": 256,
          "vsz": 512,
          "unit": "MB"
        },
        "healthStatus": "healthy"
      },
      {
        "domainID": "tenant-002",
        "sandboxID": "sb-67890",
        "status": "destroying",
        "pid": null,
        "port": null,
        "createdAt": "2026-03-14T11:00:00Z",
        "lastActiveAt": "2026-03-14T13:00:00Z",
        "idleSeconds": 3600,
        "memoryUsage": null,
        "healthStatus": "unhealthy"
      }
    ]
  }
}
```

**响应字段说明**:

| 字段 | 类型 | 说明 |
|------|------|------|
| `total` | integer | 本地沙箱总数 |
| `sandboxes` | array | 沙箱列表 |
| `sandboxes[].domainID` | string | 租户标识 |
| `sandboxes[].sandboxID` | string | 沙箱标识 |
| `sandboxes[].status` | string | 沙箱状态：`running`, `destroying`, `error` |
| `sandboxes[].pid` | integer \| null | 进程 ID，沙箱销毁后为 null |
| `sandboxes[].port` | integer \| null | 宿主机端口，沙箱销毁后为 null |
| `sandboxes[].createdAt` | string (ISO 8601) | 创建时间 |
| `sandboxes[].lastActiveAt` | string (ISO 8601) | 最后活跃时间 |
| `sandboxes[].idleSeconds` | integer | 空闲时长（秒） |
| `sandboxes[].memoryUsage` | object \| null | 内存占用，沙箱销毁后为 null |
| `sandboxes[].healthStatus` | string | 健康状态：`healthy`, `unhealthy`, `unknown` |

**行为说明**:
- 查询本地内存中所有沙箱实例
- 补充 Redis 中的元数据（创建时间、最后活跃时间）
- 实时获取进程资源占用（通过 PID）
- 执行健康检查（发送 `GET /global/health` 到沙箱）

**请求规范**:

```http

{
  "domainID": "tenant-001",
```

**请求体参数**:

**响应规范**:

```json
{
  "status": "success",
  "data": {
    "domainID": "tenant-001",

    "status": "running",
```

```json
{
  "status": "error",
  "error": {
    "code": "PORT_ALLOCATION_FAILED",
    "message": "端口分配失败，无可用端口",
    "details": {}
  }
}
```

**错误码**:

| HTTP 状态码 | 错误码 | 说明 |
|------------|--------|------|
| 400 | MISSING_DOMAIN_ID | 缺少 domainID 参数 |

- 准备 NAS 目录结构

- 生成随机密码（UUID）
- 构建 Bubblewrap 命令
- 启动 OpenCode Serve
- 写入 Redis 沙箱元数据
- 启动空闲计时器

**请求体参数**:

```json
    "portRecycled": true
  }

```

**错误响应**:

```json
{
  "status": "error",
  "error": {
    "code": "SANDBOX_NOT_FOUND",
    "message": "沙箱不存在",
    "details": {
      "domainID": "tenant-001",
      "sandboxID": "sb-99999"
    }
  }
}
```

**错误码**:

| HTTP 状态码 | 错误码 | 说明 |
|------------|--------|------|
| 400 | MISSING_DOMAIN_ID | 缺少 domainID 参数 |
| 400 | MISSING_SANDBOX_ID | 缺少 sandboxID 参数 |
| 404 | SANDBOX_NOT_FOUND | 沙箱不存在 |
| 500 | PROCESS_KILL_FAILED | 进程终止失败 |
| 500 | INTERNAL_ERROR | 内部错误 |

**行为说明**:
- 检查沙箱是否存在
- 标记沙箱状态为 `destroying`（写入 Redis）
- 终止沙箱进程（发送 SIGTERM）
- 等待进程退出（最多 10 秒）
- 强制终止（SIGKILL，若进程未退出）
- 回收端口到端口池
- 删除 Redis 沙箱记录
- 从本地内存移除沙箱实例

## 2. 接口调用流程与模块交互

本节描述每个接口触发的完整调用链，以及各模块之间的交互关系。

### 2.1 请求代理接口调用流程

**触发场景**: Echo-Agent 发送请求到 `ANY /trans/{path}`

**完整调用流程**:

```
请求到达 Engine
│
├─ 1. RequestValidator 模块
│   ├─ 提取 Headers: X-Domain-ID, X-Sandbox-ID
│   ├─ 验证参数存在性
│   └─ 返回 domainID, sandboxID
│
├─ 2. SandboxManager 模块
│   ├─ 查询本地沙箱: getSandbox(domainID, sandboxID)
│   │   ├─ 存在 → 返回沙箱实例
│   │   └─ 不存在 → 触发创建流程
│   │
│   ├─ [创建流程] createSandbox(domainID, sandboxID)
│   │   ├─ 2.1 PortAllocator 模块
│   │   │   └─ 分配端口: allocatePort()
│   │   │       ├─ 查询 Redis 获取已占用端口
│   │   │       ├─ 随机选择端口（30001-40000）
│   │   │       └─ 返回可用端口
│   │   │
│   │   ├─ 2.2 NasManager 模块
│   │   │   └─ 准备目录: prepareDirectory(domainID, sandboxID)
│   │   │       ├─ 创建 /nas/{domainID}/{sandboxID}/
│   │   │       ├─ 创建 defaultProject/
│   │   │       ├─ 创建 .config/.opencode/
│   │   │       └─ 返回 NAS 路径
│   │   │
│   │   ├─ 2.3 BubblewrapLauncher 模块
│   │   │   └─ 启动沙箱: launchSandbox(nasPath, port)
│   │   │       ├─ 生成随机密码
│   │   │       ├─ 构建 bwrap 命令
│   │   │       ├─ 启动进程
│   │   │       ├─ 等待就绪
│   │   │       └─ 返回 PID
│   │   │
│   │   ├─ 2.4 RedisClient 模块
│   │   │   └─ 写入元数据: setSandboxMetadata(domainID, sandboxID, metadata)
│   │   │       ├─ Key: sandbox:{domainID}:{sandboxID}
│   │   │       ├─ Value: {pid, port, status, createdAt, lastActiveAt}
│   │   │       └─ 返回成功
│   │   │
│   │   └─ 返回沙箱实例
│   │
│   ├─ [健康检查流程] checkSandboxHealth(sandbox)
│   │   ├─ HealthChecker 模块
│   │   │   └─ 发送 GET /global/health 到沙箱
│   │   │       ├─ 成功 → 返回 healthy
│   │   │       └─ 失败 → 触发重建
│   │   │
│   │   └─ [重建流程] rebuildSandbox(domainID, sandboxID)
│   │       ├─ 销毁旧沙箱: destroySandbox()
│   │       └─ 创建新沙箱: createSandbox()
│   │
│   └─ 返回健康沙箱实例
│
├─ 3. RequestProxy 模块
│   ├─ 路径转换: /trans/{path} → /{path}
│   ├─ Headers 转换: 移除 X-Domain-ID, X-Sandbox-ID
│   ├─ 构建请求: http://127.0.0.1:{port}/{path}
│   └─ 发送请求到沙箱
│
├─ 4. RedisClient 模块
│   └─ 更新活跃时间: updateLastActive(domainID, sandboxID)
│
└─ 5. 返回响应
    └─ 完整返回 OpenCode 原生响应
```

**关键模块交互**:
- **RequestValidator**: 参数验证和提取
- **SandboxManager**: 沙箱生命周期管理的核心协调者
- **PortAllocator**: 端口分配和回收
- **NasManager**: NAS 目录准备
- **BubblewrapLauncher**: Bubblewrap 进程启动
- **HealthChecker**: 沙箱健康检查
- **RedisClient**: Redis 状态同步
- **RequestProxy**: HTTP 请求代理

### 2.3 后台任务流程

#### 2.3.1 Engine 启动流程

**触发场景**: Engine 进程启动时

**调用流程**:

```
Engine 进程启动
│

│   └─ 初始化: initialize()
│       ├─ 1.1 加载配置: ConfigManager.loadConfig()

│       │   └─ 返回 Config 对象
│       │
│       ├─ 1.2 初始化 Redis: RedisClient.connect()
│       │   ├─ 建立 Redis 连接池
│       │   ├─ 测试连接可用性
│       │   └─ 失败则退出并报错
│       │
│       ├─ 1.3 启动 Web 服务器: FastAPI.run()
│       │   ├─ 注册 API 路由
│       │   ├─ 注册中间件（日志、异常处理）
│       │   └─ 监听端口（如 8000）
│       │
│       └─ 1.4 启动成功日志

├─ 2. TaskScheduler 模块
│   └─ 注册定时任务: registerTask()
│       ├─ 2.1 注册 HealthCheckTask
│       │   ├─ 间隔: 30 秒
│       │   └─ 返回 taskId: "health-check-001"
│       │
│       ├─ 2.2 注册 IdleCleanupTask
│       │   ├─ 间隔: 60 秒
│       │   └─ 返回 taskId: "idle-cleanup-002"
│       │
│       └─ 2.3 注册 ConfigSyncTask
│           ├─ 间隔: 300 秒
│           └─ 返回 taskId: "config-sync-003"
│
├─ 3. TaskScheduler 模块
│   └─ 启动调度器: start()
│       ├─ 创建后台线程
│       ├─ 启动所有已注册任务
│       └─ 任务按间隔自动执行
│
└─ 4. Engine 进入运行状态
    ├─ 开始接受 HTTP 请求
    ├─ 定时任务后台运行
    └─ 记录启动成功日志
```

**关键点**:
- **配置加载优先级**: 本地配置文件 → Redis 配置（启动后首次同步）
- **Redis 连接必需**: Redis 连接失败则 Engine 无法启动
- **定时任务自动运行**: 启动后立即开始执行定时任务
- **无状态设计**: Engine 启动时不恢复任何沙箱状态，沙箱按需创建

#### 2.3.2 配置同步流程

**触发场景**: ConfigSyncTask 定时任务执行（每 300 秒）

**调用流程**:

```
定时器触发（每 300 秒）
│
├─ 1. ConfigSyncTask 模块
│   └─ 执行同步: run()
│
├─ 2. RedisClient 模块
│   └─ 读取配置: getConfig("engine:config")
│       ├─ Key: engine:config (Hash)
│       ├─ 字段: maxSandboxes, idleTimeout, healthCheckInterval...
│       └─ 返回 Redis 中的配置字典
│
├─ 3. ConfigManager 模块
│   ├─ 3.1 比较配置: compareConfig(local, redis)
│   │   ├─ 遍历所有配置字段
│   │   ├─ 对比本地值和 Redis 值
│   │   └─ 返回差异字段: {maxSandboxes: 100, idleTimeout: 7200}
│   │
│   ├─ 3.2 判断是否需要更新
│   │   ├─ 无差异 → 跳过更新，记录日志，退出流程
│   │   └─ 有差异 → 继续
│   │
│   └─ 3.3 更新配置: updateConfig(changes)
│       ├─ 更新本地内存中的 Config 对象
│       ├─ 记录变更日志
│       └─ 返回更新成功
│
├─ 4. [配置变更影响处理]
│   │
│   ├─ 4.1 处理 maxSandboxes 减少
│   │   ├─ 检查当前沙箱数量: SandboxManager.count()
│   │   ├─ 若 currentCount > newMaxSandboxes
│   │   │   ├─ 计算超出数量: excess = currentCount - newMaxSandboxes
│   │   │   ├─ 查询所有沙箱空闲时间
│   │   │   ├─ 按空闲时间降序排序
│   │   │   ├─ 销毁空闲时间最长的 excess 个沙箱
│   │   │   │   └─ SandboxManager.destroySandbox(domainID, sandboxID)
│   │   │   └─ 记录销毁日志
│   │   └─ 若 currentCount <= newMaxSandboxes
│   │       └─ 无需处理
│   │
│   └─ 4.2 处理 idleTimeout 减少
│       ├─ 若 newIdleTimeout < currentIdleTimeout
│       │   ├─ 立即触发一次空闲检查
│       │   │   └─ IdleCleanupTask.run()
│       │   └─ 记录日志
│       └─ 若 newIdleTimeout >= currentIdleTimeout
│           └─ 无需处理
│
└─ 5. 记录日志
    ├─ 配置同步成功
    ├─ 变更字段列表
    └─ 影响处理结果（如销毁的沙箱数量）
```

**配置同步示例**:

```python
# Redis 配置变更
old_config = {
    "maxSandboxes": 100,
    "idleTimeout": 3600
}

new_config = {
    "maxSandboxes": 50,    # 减少最大沙箱数量
    "idleTimeout": 1800    # 减少空闲超时时长
}

# 处理流程
1. 检测到 maxSandboxes 从 100 减少到 50
2. 当前有 70 个沙箱运行中
3. 计算需要销毁 20 个沙箱
4. 按空闲时间排序，销毁空闲最长的 20 个沙箱
5. 检测到 idleTimeout 从 3600 减少到 1800
6. 立即触发空闲检查，可能销毁更多超时沙箱
```

**关键点**:
- **Redis 为准**: 配置以 Redis 为准，本地配置同步更新
- **平滑变更**: 配置变更时平滑处理，不中断服务
- **优先销毁空闲沙箱**: 减少 maxSandboxes 时优先销毁空闲时间最长的
- **立即生效**: idleTimeout 减少时立即触发检查
- **审计日志**: 记录所有配置变更和影响

### 3.2 模块调用关系详细说明

#### 3.2.1 请求代理流程调用链

```
Echo-Agent 请求
    │
    ├─→ RequestValidator.validateProxyRequest()
    │       └─→ 返回 {domainID, sandboxID}
    │
    ├─→ SandboxManager.getSandbox(domainID, sandboxID)
    │       │
    │       ├─ [沙箱不存在] → createSandbox(domainID, sandboxID)
    │       │       │
    │       │       ├─→ PortAllocator.allocatePort()
    │       │       │       └─→ 查询 Redis 已占用端口
    │       │       │       └─→ 返回可用端口
    │       │       │
    │       │       ├─→ NasManager.prepareDirectory()
    │       │       │       └─→ 创建 /nas/{domainID}/{sandboxID}
    │       │       │       └─→ 返回 nasPath
    │       │       │
    │       │       ├─→ BubblewrapLauncher.launchSandbox()
    │       │       │       └─→ 启动 bwrap 进程
    │       │       │       └─→ 返回 pid, password
    │       │       │
    │       │       └─→ RedisClient.setSandboxMetadata()
    │       │               └─→ 写入沙箱元数据
    │       │
    │       └─ [沙箱存在] → 检查健康状态
    │               └─→ HealthChecker.checkHealth()
    │                       ├─ [健康] → 返回沙箱
    │                       └─ [不健康] → rebuildSandbox()
    │                               └─→ destroySandbox() + createSandbox()
    │
    ├─→ RequestProxy.proxyRequest(sandbox, request)
    │       └─→ HTTP 转发到沙箱内 OpenCode Serve
    │       └─→ 返回响应
    │
    ├─→ RedisClient.updateLastActive(domainID, sandboxID)
    │
    └─→ ResponseBuilder.success(response)
            └─→ 返回给 Echo-Agent
```

#### 3.2.2 沙箱创建调用链

```
SandboxManager.createSandbox(domainID, sandboxID)
    │
    ├─→ PortAllocator.allocatePort()
    │       ├─ RedisClient.getAllocatedPorts()  # 查询已分配端口
    │       ├─ 随机选择端口 (30001-40000)
    │       └─ 返回 port
    │
    ├─→ NasManager.prepareDirectory(domainID, sandboxID)
    │       ├─ 创建 /nas/{domainID}/{sandboxID}/
    │       ├─ 创建 defaultProject/
    │       ├─ 创建 .config/.opencode/
    │       ├─ 设置权限 (root, rw)
    │       └─ 返回 nasPath
    │
    ├─→ BubblewrapLauncher.launchSandbox(nasPath, port)
    │       ├─ 生成随机密码 (UUID)
    │       ├─ 构建 bwrap 命令
    │       ├─ 启动进程 (subprocess.Popen)
    │       ├─ HealthChecker.checkHealth()  # 等待就绪
    │       └─ 返回 (pid, password)
    │
    └─→ RedisClient.setSandboxMetadata(domainID, sandboxID, metadata)
            ├─ Key: sandbox:{domainID}:{sandboxID}
            ├─ Value: {pid, port, status, createdAt, lastActiveAt}
            └─ 返回成功
```

#### 3.2.3 沙箱销毁调用链

```
SandboxManager.destroySandbox(domainID, sandboxID)
    │
    ├─ 获取沙箱实例
    │
    ├─→ RedisClient.updateMetadata(status="destroying")
    │       └─ 更新 Redis 状态
    │
    ├─→ BubblewrapLauncher.killSandbox(pid)
    │       ├─ 发送 SIGTERM
    │       ├─ 等待进程退出 (10 秒)
    │       ├─ [未退出] 发送 SIGKILL
    │       └─ 返回成功
    │
    ├─→ PortAllocator.recyclePort(port)
    │       └─ 从 Redis 已分配端口集合中移除
    │
    ├─→ RedisClient.deleteSandboxMetadata(domainID, sandboxID)
    │       └─ 删除 Redis 记录
    │
    └─ 从内存移除沙箱实例
```

#### 3.2.4 健康检查调用链

```
HealthCheckTask.run() [每 30 秒]
    │
    ├─→ HealthChecker.checkAllHealth(sandboxes)
    │       │
    │       └─ 对每个沙箱并发执行:
    │               └─ HTTP GET http://127.0.0.1:{port}/global/health
    │                       ├─ [成功] → 标记 healthy
    │                       └─ [失败] → 标记 unhealthy
    │
    └─→ SandboxManager.batchRebuild(unhealthySandboxes)
            └─ 对每个不健康沙箱:
                    └─ rebuildSandbox(domainID, sandboxID)
                            ├─ destroySandbox()
                            └─ createSandbox()
```

#### 3.2.5 空闲清理调用链

```
IdleCleanupTask.run() [每 60 秒]
    │
    ├─→ IdleMonitor.scanIdleSandboxes()
    │       ├─ 遍历所有沙箱
    │       ├─ 计算空闲时长 = now - lastActiveAt
    │       ├─ 筛选空闲时长 > config.idleTimeout
    │       └─ 返回超时沙箱列表
    │
    └─→ SandboxManager.batchDestroy(timeoutSandboxes)
            └─ 对每个超时沙箱:
                    └─ destroySandbox(domainID, sandboxID)
```

#### 3.2.6 配置同步调用链

```
ConfigSyncTask.run() [每 300 秒]
    │
    ├─→ RedisClient.getConfig("engine:config")
    │       └─ 返回 Redis 配置
    │
    ├─→ ConfigManager.compareConfig(localConfig, redisConfig)
    │       └─ 返回差异字段
    │
    ├─→ ConfigManager.updateConfig(changes)
    │       └─ 更新本地配置
    │
    └─ 配置变更影响处理:
            ├─ [maxSandboxes 减少]
            │       └─→ SandboxManager.count()
            │       └─→ 销毁空闲时间最长的超出沙箱
            │
            └─ [idleTimeout 减少]
                    └─→ IdleCleanupTask.run()  # 立即触发
```

```
Engine 进程启动
    │
    └─→ EngineBootstrap.initialize()
            │
            ├─→ ConfigManager.loadConfig("config.yaml")
            │       └─ 返回 Config 对象
            │
            ├─→ RedisClient.connect()
            │       └─ 建立连接池
            │
            ├─ 启动 FastAPI 应用
            │       └─ 注册路由、中间件
            │

                    └─ 启动所有定时任务
```

### 3.5 关键设计原则

#### 3.5.1 单向依赖原则

- **上层依赖下层**：API Layer → Core Layer → Infrastructure Layer
- **下层不依赖上层**：Infrastructure Layer 不依赖 Core Layer
- **避免循环依赖**：模块间调用形成有向无环图 (DAG)

#### 3.5.2 职责单一原则

- **PortAllocator**: 只负责端口分配，不关心沙箱生命周期
- **NasManager**: 只负责目录准备，不关心进程启动
- **BubblewrapLauncher**: 只负责进程管理，不关心端口分配

#### 3.5.3 核心协调者模式

- **SandboxManager** 作为唯一的核心协调者
- 所有沙箱生命周期操作都通过 SandboxManager
- 避免模块间直接耦合

#### 3.5.4 异步非阻塞设计

- **FastAPI**: 异步 Web 框架，支持高并发
- **Background Tasks**: 独立线程运行，不阻塞主线程
- **健康检查**: 并发检查所有沙箱

#### 3.4.3 BubblewrapLauncher 模块

**职责**: Bubblewrap 进程启动与管理

**输入参数**:
- `nasPath`: NAS 目录路径
- `port`: 宿主机端口

**输出参数**:
- `pid`: 进程 ID

**核心方法**:
- `launchSandbox(nasPath, port) -> (pid, password)`
  - 生成随机密码（UUID）
  - 构建 Bubblewrap 命令
  - 启动进程
  - 等待进程就绪（健康检查）
  - 返回 PID 和密码

  - 终止进程（SIGTERM）

  - 强制终止（SIGKILL，若未退出）
- `isProcessAlive(pid) -> bool`
  - 检查进程是否存活

**Bubblewrap 命令模板**:
```bash

  --unshare-pid \
  --unshare-net \
  --bind {nasPath} /sandbox \
  --ro-bind /usr /usr \
  --ro-bind /lib /lib \
  --ro-bind /lib64 /lib64 \
  --ro-bind /bin /bin \
  --dev /dev \
  --proc /proc \
  --tmpfs /tmp \
  --tmpfs /var \
  --port {hostPort}:4096 \
  --new-session \
  --die-with-parent \
  --chdir /sandbox/defaultProject \
  /usr/local/bin/opencode serve --hostname=127.0.0.1 --port=4096 --password={password}
```

- `run() -> None`
  - 调用 `HealthChecker.checkAllHealth()`
  - 对不健康沙箱调用 `SandboxManager.rebuildSandbox()`
  - 记录日志

**设计文档**: [设计文档/HealthCheckTask.md](./HealthCheckTask.md)

#### 3.7.2 IdleCleanupTask 模块

**职责**: 定时清理空闲超时的沙箱

**触发机制**: 定时任务，每 60 秒执行一次

**核心方法**:
- `run() -> None`
  - 调用 `IdleMonitor.scanIdleSandboxes()`
  - 调用 `SandboxManager.batchDestroy(超时沙箱)`
  - 记录日志

**设计文档**: [设计文档/IdleCleanupTask.md](./IdleCleanupTask.md)

#### 3.7.3 ConfigSyncTask 模块

**职责**: 定期从 Redis 同步配置到本地内存，确保 Engine 配置与 Redis 一致

**触发机制**: 定时任务，每 300 秒（5 分钟）执行一次

**核心方法**:
- `run() -> None`
  - 从 Redis 读取配置（`engine:config`）
  - 比较本地配置和 Redis 配置
  - 若 Redis 配置更新，同步到本地内存
  - 处理配置变更影响：
    - 若 `maxSandboxes` 减少，销毁超出的沙箱
    - 若 `idleTimeout` 减少，立即检查空闲沙箱
  - 记录配置同步日志

**Redis Key 设计**:
- `engine:config` (Hash): Engine 全局配置
  - `maxSandboxes`: 最大沙箱数量
  - `idleTimeout`: 空闲超时时长
  - `healthCheckInterval`: 健康检查间隔
  - 其他可动态调整的配置

**配置变更影响处理**:
```python
if newConfig.maxSandboxes < currentSandboxCount:
    # 销毁超出限制的沙箱（优先销毁空闲时间最长的）
    excess = currentSandboxCount - newConfig.maxSandboxes
    destroyIdlestSandboxes(excess)

if newConfig.idleTimeout < currentConfig.idleTimeout:
    # 立即触发一次空闲检查
    IdleCleanupTask.run()
```

**设计文档**: [设计文档/ConfigSyncTask.md](./ConfigSyncTask.md)

## 5. 错误处理与异常定义

### 5.1 异常层次结构

```
EngineException (基类)
│
├─ ValidationError           # 参数验证错误
│   ├─ MissingDomainIDError
│   └─ MissingSandboxIDError
│
├─ SandboxException          # 沙箱相关错误
│   ├─ SandboxNotFoundError
│   ├─ SandboxUnavailableError
│   ├─ PortAllocationError
│   ├─ NasPreparationError
│   └─ BubblewrapStartError
│
├─ RedisException            # Redis 相关错误
│   ├─ RedisConnectionError
│   └─ RedisOperationError
│
└─ ProxyException            # 代理相关错误
    ├─ ProxyTimeoutError
    └─ ProxyConnectionError
```

### 5.2 异常处理策略

- **ValidationError**: 返回 400 错误，记录警告日志
- **SandboxException**: 返回 503 错误，记录错误日志
- **RedisException**: 返回 503 错误，记录严重错误日志
- **ProxyException**: 返回 502/504 错误，记录错误日志
- **未知异常**: 返回 500 错误，记录严重错误日志，堆栈跟踪
