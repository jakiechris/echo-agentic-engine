# Echo-Agentic-Engine 设计文档

## 文档说明

本文档是 Engine 服务设计的核心主干，聚焦于三个部分：
1. **对外接口定义** - Engine 提供的所有 API 接口详细定义
2. **接口调用流程** - 每个接口触发的完整调用链和模块交互
3. **模块划分与职责** - 模块的功能边界、输入输出定义

具体模块的实现细节参见同级目录下的独立文档。

## 1. 对外接口定义

Engine 对外提供两类接口：
- **业务代理接口**：接收 Echo-Agent 请求，代理到沙箱内 OpenCode Serve
- **管理接口**：沙箱生命周期管理、状态查询

### 1.1 业务代理接口

#### 1.1.1 请求代理接口

**接口路径**: `ANY /trans/{path}`

**功能说明**: 将请求透传到沙箱内 OpenCode Serve，自动处理沙箱创建、健康检查、故障恢复

**请求规范**:

```http
{METHOD} /trans/{path} HTTP/1.1
Host: {engine_host}:{engine_port}
X-Domain-ID: {domainID}
X-Sandbox-ID: {sandboxID}
Content-Type: application/json
[其他 OpenCode 原生 Headers]

{OpenCode 原生请求体}
```

**路径参数**:
- `{path}`: OpenCode 原生 API 路径（如 `api/chat`, `api/files/read`）

**Headers 必需参数**:
- `X-Domain-ID` (string): 租户标识，用于存储隔离和沙箱管理
- `X-Sandbox-ID` (string): 沙箱标识，用于沙箱定位
- 其他 OpenCode 原生 Headers 完整透传

**请求体**:
- 完整透传 OpenCode 原生请求体（JSON 格式）

**响应规范**:

**成功响应** (HTTP 200):
```json
{
  "status": "success",
  "data": {OpenCode 原生响应}
}
```

**失败响应** (HTTP 4xx/5xx):
```json
{
  "status": "error",
  "error": {
    "code": "ERROR_CODE",
    "message": "错误描述",
    "details": {}
  }
}
```

**错误码定义**:

| HTTP 状态码 | 错误码 | 说明 |
|------------|--------|------|
| 400 | MISSING_DOMAIN_ID | 缺少 X-Domain-ID Header |
| 400 | MISSING_SANDBOX_ID | 缺少 X-Sandbox-ID Header |
| 503 | SANDBOX_UNAVAILABLE | 沙箱不可用（创建失败、健康检查失败） |
| 503 | SANDBOX_TIMEOUT | 请求沙箱超时 |
| 500 | INTERNAL_ERROR | Engine 内部错误 |

**行为说明**:

1. **请求验证**: 提取并验证 `X-Domain-ID`, `X-Sandbox-ID`
2. **沙箱检查**: 检查本地是否存在目标沙箱
3. **沙箱创建**: 若不存在，自动创建沙箱（准备 NAS 目录、分配端口、启动 OpenCode Serve）
4. **健康检查**: 若沙箱存在但健康检查失败，自动重建
5. **请求代理**:
   - 路径转换: `/trans/{path}` → `/{path}`
   - Headers 转换: 移除 `X-Domain-ID`, `X-Sandbox-ID`，保留其他
   - 转发到沙箱内: `http://127.0.0.1:{sandbox_port}/{path}`
6. **状态更新**: 更新沙箱活跃时间和 Redis 状态
7. **响应返回**: 完整返回 OpenCode 原生响应

**并发处理**:
- 使用 FastAPI 异步框架，每个请求独立协程处理
- 沙箱重建期间，请求在队列中排队，自动重试 3 次

### 1.2 管理接口

设计理念：所有管理接口使用 POST 方法，方法名在 URL 路径上区分，参数通过 Body 传递。

| `total` | integer | 本地沙箱总数 |
| `sandboxes` | array | 沙箱列表 |

- 查询本地内存中所有沙箱实例
- 补充 Redis 中的元数据（创建时间、最后活跃时间）
- 实时获取进程资源占用（通过 PID）
- 执行健康检查（发送 `GET /global/health` 到沙箱）

## 2. 接口调用流程与模块交互

本节描述每个接口触发的完整调用链，以及各模块之间的交互关系。

### 2.1 请求代理接口调用流程

**触发场景**: Echo-Agent 发送请求到 `ANY /trans/{path}`

**完整调用流程**:

```
请求到达 Engine
│
├─ 1. RequestValidator 模块
│   ├─ 提取 Headers: X-Domain-ID, X-Sandbox-ID
│   ├─ 验证参数存在性
│   └─ 返回 domainID, sandboxID
│
├─ 2. SandboxManager 模块
│   ├─ 查询本地沙箱: getSandbox(domainID, sandboxID)
│   │   ├─ 存在 → 返回沙箱实例
│   │   └─ 不存在 → 触发创建流程
│   │
│   ├─ [创建流程] createSandbox(domainID, sandboxID)
│   │   ├─ 2.1 PortAllocator 模块
│   │   │   └─ 分配端口: allocatePort()
│   │   │       ├─ 查询 Redis 获取已占用端口
│   │   │       ├─ 随机选择端口（30001-40000）
│   │   │       └─ 返回可用端口
│   │   │
│   │   ├─ 2.2 NasManager 模块
│   │   │   └─ 准备目录: prepareDirectory(domainID, sandboxID)
│   │   │       ├─ 创建 /nas/{domainID}/{sandboxID}/
│   │   │       ├─ 创建 defaultProject/
│   │   │       ├─ 创建 .config/.opencode/
│   │   │       └─ 返回 NAS 路径
│   │   │
│   │   ├─ 2.3 BubblewrapLauncher 模块
│   │   │   └─ 启动沙箱: launchSandbox(nasPath, port)
│   │   │       ├─ 生成随机密码
│   │   │       ├─ 构建 bwrap 命令
│   │   │       ├─ 启动进程
│   │   │       ├─ 等待就绪
│   │   │       └─ 返回 PID
│   │   │
│   │   ├─ 2.4 RedisClient 模块
│   │   │   └─ 写入元数据: setSandboxMetadata(domainID, sandboxID, metadata)
│   │   │       ├─ Key: sandbox:{domainID}:{sandboxID}
│   │   │       ├─ Value: {pid, port, status, createdAt, lastActiveAt}
│   │   │       └─ 返回成功
│   │   │
│   │   └─ 返回沙箱实例
│   │
│   ├─ [健康检查流程] checkSandboxHealth(sandbox)
│   │   ├─ HealthChecker 模块
│   │   │   └─ 发送 GET /global/health 到沙箱
│   │   │       ├─ 成功 → 返回 healthy
│   │   │       └─ 失败 → 触发重建
│   │   │
│   │   └─ [重建流程] rebuildSandbox(domainID, sandboxID)
│   │       ├─ 销毁旧沙箱: destroySandbox()
│   │       └─ 创建新沙箱: createSandbox()
│   │
│   └─ 返回健康沙箱实例
│
├─ 3. RequestProxy 模块
│   ├─ 路径转换: /trans/{path} → /{path}
│   ├─ Headers 转换: 移除 X-Domain-ID, X-Sandbox-ID
│   ├─ 构建请求: http://127.0.0.1:{port}/{path}
│   └─ 发送请求到沙箱
│
├─ 4. RedisClient 模块
│   └─ 更新活跃时间: updateLastActive(domainID, sandboxID)
│
└─ 5. 返回响应
    └─ 完整返回 OpenCode 原生响应
```

**关键模块交互**:
- **RequestValidator**: 参数验证和提取
- **SandboxManager**: 沙箱生命周期管理的核心协调者
- **PortAllocator**: 端口分配和回收
- **NasManager**: NAS 目录准备
- **BubblewrapLauncher**: Bubblewrap 进程启动
- **HealthChecker**: 沙箱健康检查
- **RedisClient**: Redis 状态同步
- **RequestProxy**: HTTP 请求代理

#### 2.3.2 配置同步流程

**触发场景**: ConfigSyncTask 定时任务执行（每 300 秒）

**调用流程**:

```
定时器触发（每 300 秒）
│
├─ 1. ConfigSyncTask 模块
│   └─ 执行同步: run()
│
├─ 2. RedisClient 模块
│   └─ 读取配置: getConfig("engine:config")
│       ├─ Key: engine:config (Hash)
│       ├─ 字段: maxSandboxes, idleTimeout, healthCheckInterval...
│       └─ 返回 Redis 中的配置字典
│
├─ 3. ConfigManager 模块
│   ├─ 3.1 比较配置: compareConfig(local, redis)
│   │   ├─ 遍历所有配置字段
│   │   ├─ 对比本地值和 Redis 值
│   │   └─ 返回差异字段: {maxSandboxes: 100, idleTimeout: 7200}
│   │
│   ├─ 3.2 判断是否需要更新
│   │   ├─ 无差异 → 跳过更新，记录日志，退出流程
│   │   └─ 有差异 → 继续
│   │
│   └─ 3.3 更新配置: updateConfig(changes)
│       ├─ 更新本地内存中的 Config 对象
│       ├─ 记录变更日志
│       └─ 返回更新成功
│
├─ 4. [配置变更影响处理]
│   │
│   ├─ 4.1 处理 maxSandboxes 减少
│   │   ├─ 检查当前沙箱数量: SandboxManager.count()
│   │   ├─ 若 currentCount > newMaxSandboxes
│   │   │   ├─ 计算超出数量: excess = currentCount - newMaxSandboxes
│   │   │   ├─ 查询所有沙箱空闲时间
│   │   │   ├─ 按空闲时间降序排序
│   │   │   ├─ 销毁空闲时间最长的 excess 个沙箱
│   │   │   │   └─ SandboxManager.destroySandbox(domainID, sandboxID)
│   │   │   └─ 记录销毁日志
│   │   └─ 若 currentCount <= newMaxSandboxes
│   │       └─ 无需处理
│   │
│   └─ 4.2 处理 idleTimeout 减少
│       ├─ 若 newIdleTimeout < currentIdleTimeout
│       │   ├─ 立即触发一次空闲检查
│       │   │   └─ IdleCleanupTask.execute()
│       │   └─ 记录日志
│       └─ 若 newIdleTimeout >= currentIdleTimeout
│           └─ 无需处理
│
└─ 5. 记录日志
    ├─ 配置同步成功
    ├─ 变更字段列表
    └─ 影响处理结果（如销毁的沙箱数量）
```

**配置同步示例**:

```python
# Redis 配置变更
old_config = {
    "maxSandboxes": 100,
    "idleTimeout": 3600
}

new_config = {
    "maxSandboxes": 50,    # 减少最大沙箱数量
    "idleTimeout": 1800    # 减少空闲超时时长
}

# 处理流程
1. 检测到 maxSandboxes 从 100 减少到 50
2. 当前有 70 个沙箱运行中
3. 计算需要销毁 20 个沙箱
4. 按空闲时间排序，销毁空闲最长的 20 个沙箱
5. 检测到 idleTimeout 从 3600 减少到 1800
6. 立即触发空闲检查，可能销毁更多超时沙箱
```

**关键点**:
- **Redis 为准**: 配置以 Redis 为准，本地配置同步更新
- **平滑变更**: 配置变更时平滑处理，不中断服务
- **优先销毁空闲沙箱**: 减少 maxSandboxes 时优先销毁空闲时间最长的
- **立即生效**: idleTimeout 减少时立即触发检查
- **审计日志**: 记录所有配置变更和影响

### 3.5 关键设计原则

#### 3.5.1 单向依赖原则

- **上层依赖下层**：API Layer → Core Layer → Infrastructure Layer
- **下层不依赖上层**：Infrastructure Layer 不依赖 Core Layer
- **避免循环依赖**：模块间调用形成有向无环图 (DAG)

#### 3.5.2 职责单一原则

- **PortAllocator**: 只负责端口分配，不关心沙箱生命周期
- **NasManager**: 只负责目录准备，不关心进程启动
- **BubblewrapLauncher**: 只负责进程管理，不关心端口分配

#### 3.5.3 核心协调者模式

- **SandboxManager** 作为唯一的核心协调者
- 所有沙箱生命周期操作都通过 SandboxManager
- 避免模块间直接耦合

#### 3.5.4 异步非阻塞设计

- **FastAPI**: 异步 Web 框架，支持高并发
- **Background Tasks**: 独立线程运行，不阻塞主线程
- **健康检查**: 并发检查所有沙箱

Engine 采用分层架构设计，共 7 层 23 个模块：

```
Echo-Agentic-Engine
│
├─ Router Layer (路由层)
│   ├─ ProxyRouter              # OpenCode API 代理请求路由
│   ├─ ListSandboxRouter        # 列出所有沙箱路由

│   ├─ CreateSandboxRouter      # 创建沙箱路由
│   └─ DestroySandboxRouter     # 销毁沙箱路由
│
├─ API Layer (接口层)
│   ├─ RequestValidator         # 请求验证与参数提取
│   └─ ResponseBuilder          # 响应构建与错误处理
│
├─ Core Layer (核心层)
│   ├─ SandboxManager           # 沙箱生命周期管理（核心协调者）
│   ├─ RequestProxy             # HTTP 请求代理

│
├─ Infrastructure Layer (基础设施层)
│   ├─ PortAllocator            # 端口分配与回收
│   ├─ NasManager               # NAS 目录管理
│   ├─ BubblewrapLauncher       # Bubblewrap 进程启动
│   └─ RedisClient              # Redis 客户端
│
├─ Monitoring Layer (监控层)
│   ├─ HealthChecker            # 沙箱健康检查

│   ├─ IdleMonitor              # 空闲超时监控
│   └─ LogReader                # 日志读取
│
├─ Bootstrap Layer (启动层)
│   ├─ EngineBootstrap          # Engine 启动初始化
│   └─ TaskScheduler            # 定时任务调度器
│
└─ Background Tasks (后台任务层)
    ├─ HealthCheckTask          # 健康检查巡检任务
    ├─ IdleCleanupTask          # 空闲沙箱清理任务
    └─ ConfigSyncTask           # 配置同步任务
```

**分层职责说明**：

| 层级 | 职责 | 调用关系 |
|------|------|----------|
| Router Layer | FastAPI 路由函数,接收 HTTP 请求并协调模块调用 | 被 FastAPI 框架调用 |
| API Layer | 处理 HTTP 请求/响应，参数验证，响应格式化 | 被 Router Layer 调用 |
| Core Layer | 核心业务逻辑，协调各模块完成业务流程 | 被 Router Layer/API Layer 调用 |
| Infrastructure Layer | 提供基础设施服务（端口、目录、进程、存储） | 被 Core Layer 调用 |
| Monitoring Layer | 监控沙箱状态、资源、健康、日志 | 被 Router Layer/Core Layer/API Layer 调用 |
| Bootstrap Layer | Engine 启动初始化，定时任务调度 | Engine 进程启动时执行 |
| Background Tasks | 后台定时任务，巡检和清理 | 由 TaskScheduler 定时触发 |

- `validateAdminCreateRequest(body) -> {domainID, sandboxID}`
  - 从 Body 提取 domainID, sandboxID
  - 验证参数存在性
  - 返回 domainID, sandboxID
- `validateAdminDestroyRequest(body) -> {domainID, sandboxID}`
  - 从 Body 提取 domainID, sandboxID
  - 验证参数存在性
  - 返回 domainID, sandboxID

### 4.3 Core Layer (核心层)

#### 4.3.1 SandboxManager 模块

**职责**: 沙箱生命周期管理的核心协调者，协调各模块完成沙箱的创建、销毁、重建

**输入参数**:
- `domainID`: 租户标识
- `sandboxID`: 沙箱标识
- `sandbox`: 沙箱实例对象

**输出参数**:
- `Sandbox`: 沙箱实例对象
- `SandboxList`: 沙箱列表

**核心方法**:
- `getSandbox(domainID, sandboxID) -> Sandbox | None`
  - 从内存查找沙箱实例
  - 不存在则返回 None
  - 存在则调用 HealthChecker 检查健康
  - 不健康则调用 rebuildSandbox 重建
  - 返回沙箱实例
- `listAllSandboxes() -> List[Sandbox]`
  - 从内存获取所有沙箱实例
  - 返回沙箱列表
- `createSandbox(domainID, sandboxID) -> Sandbox`
  - 调用 PortAllocator.allocatePort() 分配端口
  - 调用 NasManager.prepareDirectory() 准备目录
  - 调用 BubblewrapLauncher.launchSandbox() 启动进程
  - 调用 RedisClient.setSandboxMetadata() 写入元数据
  - 返回沙箱实例
- `destroySandbox(domainID, sandboxID) -> bool`
  - 调用 RedisClient.updateMetadata(status="destroying") 更新状态
  - 调用 BubblewrapLauncher.killSandbox() 终止进程
  - 调用 PortAllocator.recyclePort() 回收端口
  - 调用 RedisClient.deleteSandboxMetadata() 删除元数据
  - 从内存移除沙箱实例
  - 返回成功
- `rebuildSandbox(domainID, sandboxID) -> Sandbox`
  - 调用 destroySandbox() 销毁旧沙箱
  - 调用 createSandbox() 创建新沙箱
  - 返回新沙箱实例
- `batchRebuild(sandboxes) -> List[Sandbox]`
  - 批量重建沙箱
  - 对每个沙箱调用 rebuildSandbox()
  - 返回重建后的沙箱列表
- `batchDestroy(sandboxes) -> List[bool]`
  - 批量销毁沙箱
  - 对每个沙箱调用 destroySandbox()
  - 返回销毁结果列表

**参与流程**:
- 流程 3.2.1: OpenCode API 代理主流程
- 流程 3.2.2: 管理接口：列出所有沙箱
- 流程 3.2.3: 管理接口：查询单个沙箱
- 流程 3.2.4: 管理接口：创建沙箱
- 流程 3.2.5: 管理接口：销毁沙箱
- 流程 3.2.7: HealthCheckTask 调用链
- 流程 3.2.8: IdleCleanupTask 调用链

**依赖模块**:

- `BubblewrapLauncher`: 进程启动
- `RedisClient`: Redis 状态同步

### 4.4 Infrastructure Layer (基础设施层)

#### 4.4.1 PortAllocator 模块

**职责**: 端口分配与回收管理

**输入参数**:
- `port`: 端口号

**输出参数**:
- `port`: 分配的端口号

**核心方法**:
- `allocatePort() -> int`
  - 调用 RedisClient.queryAllocatedPorts() 查询已占用端口
  - 在端口范围 30001-40000 内随机选择端口
  - 避开已占用端口
  - 返回可用端口
- `recyclePort(port) -> bool`
  - 从 Redis 已分配端口集合中移除
  - 返回成功
- `queryAllocatedPorts() -> List[int]`
  - 查询 Redis 获取所有已分配端口
  - 返回端口列表

**输出参数**: 无

## 5. 系统配置

### 5.1 配置项定义

Engine 配置存储在 Redis `engine:config` Key (Hash 结构),包含以下字段:

| 配置项 | 类型 | 默认值 | 说明 | 作用域 |
|--------|------|--------|------|--------|
| **portRange** | Dict | `{min: 30001, max: 40000}` | 端口池范围 | PortAllocator |
| **idleTimeout** | int | 3600 | 空闲超时时长 (秒) | IdleMonitor |
| **healthCheckInterval** | int | 30 | 健康检查间隔 (秒) | HealthCheckTask |
| **idleCheckInterval** | int | 60 | 空闲检查间隔 (秒) | IdleCleanupTask |
| **configSyncInterval** | int | 300 | 配置同步间隔 (秒) | ConfigSyncTask |
| **maxRetries** | int | 3 | 请求重试次数 | 全局 |
| **healthCheckTimeout** | int | 5 | 健康检查超时 (秒) | HealthChecker |
| **maxSandboxes** | int | 50 | 最大沙箱数量 | SandboxManager |
| **redisHost** | str | "localhost" | Redis 主机地址 | RedisClient |
| **redisPort** | int | 6379 | Redis 端口 | RedisClient |
| **redisPassword** | str | "" | Redis 密码 (可选) | RedisClient |
| **redisDB** | int | 0 | Redis 数据库编号 | RedisClient |
| **nasRootPath** | str | "/nas" | NAS 根路径 | NasManager |
| **enginePort** | int | 8000 | Engine 监听端口 | EngineBootstrap |
| **openCodeCommand** | str | "/usr/local/bin/opencode" | OpenCode 命令路径 | BubblewrapLauncher |
| **logLevel** | str | "INFO" | 日志级别 | 全局 |

### 5.2 配置加载机制

**初始化加载** (流程 3.2.6):

```
EngineBootstrap.initialize()
  └─ ConfigManager.loadConfig()
      ├─ RedisClient.fetchConfig()  # 从 Redis 读取配置
      ├─ 若 Redis 中无配置,使用内置默认值
      └─ 验证配置项合法性
```

**配置同步** (流程 3.2.9):

```
ConfigSyncTask.execute()  # 每 300 秒执行
  ├─ RedisClient.fetchConfig()  # 读取最新配置
  ├─ ConfigManager.compareConfig(local, redis)  # 比较差异
  ├─ ConfigManager.updateConfig(changes)  # 更新内存配置
  └─ 处理配置变更影响
      ├─ maxSandboxes 减少 → 销毁空闲时间最长的沙箱
      └─ idleTimeout 减少 → 立即触发 IdleCleanupTask
```

### 5.3 配置访问方式

所有模块通过 `ConfigManager` 访问配置,每次从内存读取最新值:

```python
class SandboxManager:
    def createSandbox(self, domainID, sandboxID):
        config = ConfigManager.get_instance().getConfig()
        max_sandboxes = config['maxSandboxes']

        if len(self.listAllSandboxes()) >= max_sandboxes:
            raise MaxSandboxesExceededError(...)
```

### 5.4 配置变更影响

| 配置项变更 | 影响处理 | 触发模块 |
|-----------|---------|---------|
| **maxSandboxes 减少** | 销毁空闲时间最长的沙箱,直到沙箱数量不超过新上限 | ConfigSyncTask |
| **idleTimeout 减少** | 立即触发 IdleCleanupTask.execute() | ConfigSyncTask |
| **其他配置变更** | 仅更新内存配置,无需额外处理 | ConfigSyncTask |

### 5.2 异常处理策略

- **ValidationError**: 返回 400 错误，记录警告日志
- **SandboxException**: 返回 503 错误，记录错误日志
- **RedisException**: 返回 503 错误，记录严重错误日志
- **ProxyException**: 返回 502/504 错误，记录错误日志
- **未知异常**: 返回 500 错误，记录严重错误日志，堆栈跟踪

### 7.2 依赖注入设计

使用依赖注入容器管理模块实例和依赖关系。

#### 7.2.1 依赖注入容器

```python
from typing import Dict, Type, Any

class DIContainer:
    """依赖注入容器"""

    def __init__(self):
        self._instances: Dict[Type, Any] = {}
        self._factories: Dict[Type, callable] = {}

    def register_instance(self, interface: Type, instance: Any) -> None:
        """注册单例实例"""
        self._instances[interface] = instance

    def register_factory(self, interface: Type, factory: callable) -> None:
        """注册工厂函数"""
        self._factories[interface] = factory

    def resolve(self, interface: Type) -> Any:
        """解析依赖"""
        if interface in self._instances:
            return self._instances[interface]

        if interface in self._factories:
            instance = self._factories[interface](self)
            self._instances[interface] = instance
            return instance

        raise ValueError(f"Dependency not registered: {interface}")
```

#### 7.2.2 容器配置示例

```python
def configure_container() -> DIContainer:
    """配置依赖注入容器"""
    container = DIContainer()

    # Infrastructure Layer
    container.register_factory(IPortAllocator, lambda c: PortAllocator(
        redis_client=c.resolve(IRedisClient)
    ))

    container.register_factory(INasManager, lambda c: NasManager(
        nas_root_path=c.resolve(IConfigManager).get('nas_root_path')
    ))

    container.register_factory(IBubblewrapLauncher, lambda c: BubblewrapLauncher(
        open_code_command=c.resolve(IConfigManager).get('open_code_command')
    ))

    container.register_factory(IRedisClient, lambda c: RedisClient(
        host=c.resolve(IConfigManager).get('redis_host'),
        port=c.resolve(IConfigManager).get('redis_port')
    ))

    # Monitoring Layer
    container.register_factory(IHealthChecker, lambda c: HealthChecker(
        timeout=c.resolve(IConfigManager).get('health_check_timeout')
    ))

    container.register_instance(IResourceMonitor, ResourceMonitor())
    container.register_instance(IIdleMonitor, IdleMonitor())
    container.register_instance(ILogReader, LogReader())

    # Core Layer
    container.register_factory(ISandboxManager, lambda c: SandboxManager(
        port_allocator=c.resolve(IPortAllocator),
        nas_manager=c.resolve(INasManager),
        bubblewrap_launcher=c.resolve(IBubblewrapLauncher),
        redis_client=c.resolve(IRedisClient),
        health_checker=c.resolve(IHealthChecker)
    ))

    container.register_instance(IRequestProxy, RequestProxy())
    container.register_factory(IConfigManager, lambda c: ConfigManager(
        redis_client=c.resolve(IRedisClient)
    ))

    return container
```
