## 0. 产品形态与文件管理

### 0.1 产品形态

**Web UI 界面布局**:
```
┌─────────────────────────────────────────────────────────────┐
│                     Web UI (由 Echo-Agent 实现)               │
│                                                               │
│  ┌─────────┐   ┌──────────────────┐   ┌─────────────────┐   │
│  │ 左侧栏  │   │    中间区域      │   │    右侧栏       │   │
│  │         │   │                  │   │                 │   │
│  │ Session │   │  OpenCode对话    │   │  文件管理       │   │
│  │  列表   │   │     窗口         │   │  (文件树/编辑)  │   │
│  │         │   │                  │   │                 │   │
│  │ - sess1 │   │  [对话消息区域]  │   │  /              │   │
│  │ - sess2 │   │                  │   │  ├─ defaultProject/│ │

│  │         │   │                  │   │  │  ├─ docs/     │   │

└─────────────────────────────────────────────────────────────┘
```

**关键特性**:
- **左侧 Session 列表**: 显示当前 sandbox 下所有 session（由 OpenCode Serve 管理）
- **中间对话窗口**: 对应某个 OpenCode Serve 沙箱的对话界面
- **右侧文件管理**: 展示该 sandbox 对应的 NAS 文件目录

**用户视角的文件路径**:
- 用户在 Web UI 看到的根目录 `/` = 实际 NAS 路径 `/nas/{domainID}/{sandboxID}/`
- 用户只能看到自己 sandbox 范围内的文件，无法访问其他 sandbox
- 用户能看到 `defaultProject` 文件夹（当前唯一 project）

**未来扩展**:
- 一个沙箱可包含多个 project 文件夹（如 `project1`, `project2`）
- 每个 project 下可管理多个 session
- 当前只支持 `defaultProject` 一个 project

### 0.2 文件管理系统

**架构设计原则**:
- **完全解耦**: 文件管理与 OpenCode Serve、Engine 完全独立
- **Echo-Agent 负责**: 文件管理由 Echo-Agent 实现，不依赖 Engine
- **NAS 路径隔离**: Echo-Agent 根据 domainID 和 sandboxID 映射 NAS 路径

**文件管理流程**:
```
用户在 Web UI 操作文件
  ↓ 请求到 Echo-Agent
Echo-Agent 解析用户身份
  ├─ 获取 domainID, sandboxID
  ├─ 映射 NAS 路径: /nas/{domainID}/{sandboxID}/
  └─ 执行文件操作（读写、上传下载等）
  ↓ 返回结果给 Web UI
```

**文件操作接口**（由 Echo-Agent 提供，不在本文档范围内）:
- 文件列表查询
- 文件上传/下载
- 文件创建/删除/重命名
- 文件编辑
- 目录管理

**配置管理**:
- OpenCode 配置文件（如 `opencode.json`）通过文件管理系统维护
- 用户可通过 Web UI 编辑配置文件
- Engine 不负责配置管理，只负责沙箱生命周期

## 1. 整体架构

```

├─ OpenCode UI (聊天交互界面)
└─ 文件管理 Web UI

服务网关层
└─ Echo-Agent (请求路由 + 文件服务)

沙箱管理层
└─ Echo-Agentic-Engine (本系统)
```

| **OpenCode UI** | 用户聊天交互界面 | 不感知沙箱,仅通过 echo-agent 通信 |

## 2. Echo-Agentic-Engine 核心功能

### 2.1 请求路由
- 每个请求通过 `domainID` + `sandboxID` 唯一标识沙箱
- 从 HTTP Headers 提取 `X-Domain-ID` 和 `X-Sandbox-ID`
- 白名单验证由上游网关负责，本系统默认 domainID 合法

### 2.2 反向代理

Echo-Agentic-Engine 对外暴露统一代理路径，所有请求通过 header 携带标识信息：

**对外接口**：
```
ANY  /trans/{path}                   # 统一代理路径（path 为 OpenCode Serve 原生路径）
```

**请求标识**（HTTP Headers）：
- `X-Domain-ID`: 租户标识（用于白名单验证和存储隔离）
- `X-Sandbox-ID`: 沙箱标识（用于定位具体沙箱实例）

**路由规则**：
```
客户端请求:  POST /trans/api/chat
Headers:     X-Domain-ID: tenant-001
            X-Sandbox-ID: sandbox-abc
            (其他原有headers)

转发目标:    POST http://127.0.0.1:{port}/api/chat
Headers:     (保留所有原始headers，移除 X-Domain-ID 和 X-Sandbox-ID)
```

**支持的所有 OpenCode Serve 原生接口**：
- HTTP REST API：对话、文件操作、配置管理等
- 支持所有 HTTP 方法和协议（GET、POST、PUT、DELETE等）
- 透传所有请求头和请求体

**关键特性**：
- ✅ 路径透明映射：去掉 `/trans` 前缀后原样透传
- ✅ Header 处理：提取 `domainID`/`sandboxID` 用于路由，不传递给 OpenCode
- ✅ 协议无关：支持 HTTP、SSE、WebSocket 等所有协议
- ✅ 无业务逻辑：不修改请求/响应内容，纯透传

### 2.3 沙箱管理

**生命周期管理职责**:
- Engine 完全负责沙箱的创建、监控、销毁
- 上游网关（Echo-Agent）只负责路由，不感知沙箱生命周期
- 沙箱自动销毁机制：长时间无请求自动清理

**创建沙箱**
```bash
bwrap \
  # ── 系统目录只读 ──
  --ro-bind /usr /usr \
  --ro-bind /etc/resolv.conf /etc/resolv.conf \
  --ro-bind /etc/ssl /etc/ssl \
  --ro-bind /lib /lib \          # 条件挂载
  --ro-bind /lib64 /lib64 \      # 条件挂载
  --ro-bind /bin /bin \           # 条件挂载
  --ro-bind /sbin /sbin \         # 条件挂载
  --ro-bind /proc /proc \
  --dev /dev \
  #
  # ── NAS 数据挂载（关键映射）──
  --bind /nas/{domainID}/{sandboxID} /sandbox \
  --bind /nas/{domainID}/{sandboxID}/defaultProject /sandbox/defaultProject \
  #
  # ── 命名空间隔离 ──
  --unshare-user \
  --unshare-pid \
  --unshare-net \                # 隔离网络命名空间
  --die-with-parent \
  --new-session \
  #
  # ── 端口映射 ──
  --port {分配的宿主机端口}:4096 \  # 如 --port 30001:4096
  #
  # ── 工作目录与环境变量 ──
  --chdir /sandbox/defaultProject \
  --setenv PATH "/usr/local/bin:/usr/bin:/bin" \
  --setenv HOME "/tmp" \
  --setenv XDG_DATA_HOME "/sandbox/data" \
  --setenv XDG_CONFIG_HOME "/sandbox/config" \
  --setenv XDG_STATE_HOME "/sandbox/data/state" \
  --setenv XDG_CACHE_HOME "/sandbox/data/cache" \
  --setenv OPENCODE_SERVER_USERNAME "opencode" \
  --setenv OPENCODE_SERVER_PASSWORD "{随机 UUID}" \
  #
  # ── 启动 opencode ──
  -- opencode serve \
  --hostname=127.0.0.1 \
  --port=4096    # 沙箱内固定监听 4096 端口
```

**端口与网络说明**:
- **网络模式**: 使用 `--unshare-net` 隔离网络命名空间
- **端口映射**: 使用 `--port {宿主机端口}:4096` 映射端口（如 `--port 30001:4096`）
- **端口映射机制**: 类似 Docker 的 `-p` 参数，将宿主机端口映射到沙箱内端口
  - Echo-Agent 访问 Engine: `{engine_ip}:8080`
  - Engine 访问沙箱: `127.0.0.1:{宿主机端口}` (范围: 30001-40000)
  - 沙箱内 OpenCode Serve 监听: `127.0.0.1:4096`
- **访问方式**:
  ```
  Echo-Agent 请求
    → Engine:{宿主机IP}:8080
    → Engine 转发到 127.0.0.1:{sandbox_port}
    → Bubblewrap 端口映射 (--port 30001:4096)
    → 沙箱内 OpenCode Serve (127.0.0.1:4096)
  ```
- **网络访问**: 沙箱内 OpenCode Serve 需要访问外部 AI API 时，内网环境已具有网络通路
- **优势**:
  - 网络隔离，安全性更高
  - 通过端口映射精确控制访问
  - 避免沙箱间网络干扰

**Redis 记录**:
- `sandbox_port`: 宿主机映射端口（30001, 30002...）

关键参数：

| 参数 | 作用 |
|------|------|
| `--ro-bind src dst` | 只读挂载，沙箱内无法修改系统文件 |
| `--bind src dst` | 可读写挂载，用户数据目录 |
| `--dev /dev` | 仅生成 null/zero/full/random/urandom/tty |
| `--unshare-user` | 用户命名空间，无需 root |
| `--unshare-pid` | PID 命名空间，看不到其他进程 |
| `--die-with-parent` | Gateway 退出时内核自动 SIGKILL 子进程 |
| `--new-session` | 独立进程组，信号不跨沙箱传播 |
| `--port {host}:4096` | 端口映射，沙箱内 4096 端口映射到宿主机端口 |

**关键参数说明**:
- `--bind /nas/{domainID}/{sandboxID} /sandbox`: NAS 存储映射到沙箱根目录
- `--bind` vs `--ro-bind`: NAS 使用读写挂载，系统目录只读
- `--die-with-parent`: Echo-Agentic-Engine 崩溃时自动清理沙箱
- `--new-session`: 安全隔离，防止终端劫持
- `--unshare-net`: 隔离网络命名空间，提升安全性
- `--port {宿主机端口}:4096`: 端口映射，将沙箱内 4096 端口映射到宿主机端口
- OpenCode Serve 在沙箱内固定监听 `127.0.0.1:4096`

**生命周期管理**:
- **创建并启动（原子操作）**:
  - Engine 收到请求时检查本地沙箱是否存在
  - 不存在则创建并启动 OpenCode Serve
  - 创建成功后写入 Redis: `sandbox:{domainID}:{sandboxID}`
- **运行**:
  - 每次收到该沙箱的请求，重置空闲计时器
  - 定期轮询沙箱健康状态（`/global/health` 接口，30秒超时）
  - 定时向 Redis 更新心跳状态
- **停止并销毁（原子操作）**:
  - 空闲超时自动触发：定时器监控沙箱空闲时长（默认 30 分钟无请求）
  - 或管理接口手动触发：`POST /admin/sandbox/destroy` (body: `{"domainID": "xxx", "sandboxID": "xxx"}`)
  - 操作流程：
    1. 写入 Redis 标记 `status: "destroying"`
    2. 杀掉沙箱进程
    3. 回收端口等资源
    4. 从 Redis 删除沙箱记录

### 2.4 空闲计时器机制

**设计目标**:
- 自动回收长期无请求的沙箱，释放资源
- 避免沙箱无限累积导致资源耗尽
- 支持用户长时间交互后的会话恢复

**实现机制**:
```
每个沙箱维护独立的计时器:
┌─────────────────────────────────────┐
│  Sandbox Instance                    │
│                                      │
│  • idleTimer: Timer                  │
│  • idleTimeout: 30min (可配置)        │
│  • lastActiveAt: timestamp           │
│                                      │
│  ResetIdleTimer():                   │
│    - 停止旧计时器                     │
│    - 启动新计时器(idleTimeout)        │
│    - 更新 lastActiveAt               │
│                                      │
│  OnTimeout():                        │
│    - 标记 Redis: status="destroying" │
│    - 停止并销毁沙箱                  │
│    - 删除 Redis 记录                 │
└─────────────────────────────────────┘
```

**关键流程**:
1. **沙箱创建时**: 启动空闲计时器，设置超时回调
2. **请求到达时**: 调用 `ResetIdleTimer()` 重置计时器
3. **超时触发时**:
   - 标记 Redis: `status = "destroying"`
   - 停止并销毁沙箱（杀掉进程、回收端口）
   - 删除 Redis 记录

**配置参数**:
- `idleTimeout`: 空闲超时时长（默认 1小时，可在 Web UI 配置）
- `heartbeatTimeout`: Engine 与沙箱心跳超时时长（默认 30秒）
- 配置来源: Echo-Agent 设置到 Redis，Engine 读取使用

**注意事项**:
- 每个沙箱独立计时器，互不影响
- 计时器精度要求不高，允许 ±1 分钟误差
- 销毁操作需要考虑优雅关闭（可选）

### 2.5 状态同步

**同步时机**:
| 事件 | 同步内容 | 目标 |
|------|---------|------|
| 沙箱创建 | 沙箱信息 + 本机 IP + 创建时间 | Redis |
| 沙箱请求 | 更新最后活跃时间 | Redis |
| 沙箱空闲超时 | 标记 `status: "destroying"` | Redis |
| 沙箱销毁完成 | 删除沙箱记录 | Redis |
| 心跳上报 | 节点负载 + 时间戳 | Redis |

**Redis 数据结构**:
```
# 沙箱元数据 (Hash)
sandbox:{domainID}:{sandboxID} = {
    "engine_ip": "x.x.x.x",
    "engine_port": 8080,
    "sandbox_port": 30001,
    "status": "running",          # running / destroying
    "pid": 12345,
    "created_at": "2026-03-13T10:00:00Z",
    "last_active_at": "2026-03-13T10:30:00Z"  # 最后活跃时间
}

# 关键说明:
# - engine_port: Echo-Agentic-Engine 对外服务端口（固定 8080）
# - sandbox_port: Bubblewrap 端口映射的宿主机端口（动态分配，如 30001, 30002...）
# - 沙箱内 OpenCode Serve 固定监听 127.0.0.1:4096
# - 使用 --unshare-net 隔离网络，--port 映射端口
# - last_active_at: 每次请求更新，用于上游网关判断沙箱存活

# 节点信息 (Hash)
engine:nodes = {
    "10.0.1.1:8080": {"capacity": 500, "running": 123, "heartbeat": 1709520060},
    "10.0.1.2:8080": {"capacity": 500, "running": 89, "heartbeat": 1709520061}
}

# 白名单 (Set)
whitelist:domainIDs = ["domain1", "domain2"]

# 配置参数 (Hash)
config = {
    "sandbox_idle_timeout": "3600",           # 沙箱空闲超时（秒），默认 1小时
    "engine_sandbox_heartbeat_timeout": "30", # Engine与沙箱心跳超时（秒），默认 30秒
    "echo_agent_engine_heartbeat_timeout": "30", # Echo-Agent与Engine心跳超时（秒），默认 30秒
    "max_sandboxes_per_engine": "50"          # 单Engine最大沙箱数，默认 50
}
```

**心跳机制**:
```
Echo-Agentic-Engine 节点 ──(每 10s)──→ Redis 更新 heartbeat

上游网关检查逻辑:
if (now - heartbeat) > 30s:
    标记节点离线
    从可用列表移除
```
- 沙箱创建后,将信息 + 本机 IP 写入 redis
- 沙箱存活状态实时同步
- Session 信息同步（待实现）

**配置同步机制**:
```
Engine 启动时:
  └─ 从 Redis 读取所有配置 → 加载到本地内存

Engine 运行时:
  ├─ 定时轮询（每 10s）: Redis config → 本地内存
  └─ 使用配置时从本地内存读取（高性能）

配置更新流程:
  Web UI 修改 → Echo-Agent 写入 Redis config → Engine 轮询同步到内存 → 生效
```
- 避免每次请求都查询 Redis，提升性能
- 支持动态调整配置，无需重启 Engine
- 配置项包括：超时时长、并发限制等

**配置更新生效策略**:
- **max_sandboxes_per_engine 更新**: 立即生效，若当前沙箱数超过新上限，多余沙箱直接销毁（空闲超时优先销毁）
- **sandbox_idle_timeout 更新**: 立即生效，已运行沙箱的空闲计时器立即重置使用新值
- **其他超时配置更新**: 立即生效，适用于所有沙箱

#### 4.2.3 故障恢复流程

**故障场景说明**:
- Redis 记录显示沙箱存活（`status: "running"`, `last_active_at` 近期更新）
- 但实际沙箱进程已崩溃或异常，无法处理请求
- 可能原因：
  - OpenCode Serve 进程崩溃
  - 沙箱端口被占用
  - 资源耗尽（内存、磁盘）
  - 进程僵死但未退出

**检测时机**:
- Engine 定期轮询发送 `GET /global/health` 请求到沙箱内 OpenCode Serve
- **健康检测接口**: `GET /global/health`
- **响应格式**: `{ "healthy": true, "version": "string" }`
- 心跳超时时长：30秒
- 检测失败（连接拒绝/超时）→ 触发自动重建
- 或进程监控发现沙箱进程已退出

**恢复流程**:
```
步骤 1: Engine 检测到沙箱异常
├─ 请求到达时，尝试连接沙箱端口
├─ 连接失败（Connection refused / timeout）
├─ 检查进程是否存在（通过 PID）
└─ 确认沙箱不可用

步骤 2: 自动重建沙箱
├─ 清理残留资源（如有）
│   ├─ 杀掉残留进程
│   ├─ 回收端口
│   └─ 更新 Redis: status = "destroying"
├─ 基于 domainID + sandboxID 准备 NAS 目录
│   └─ NAS 数据自动恢复 (共享存储)
├─ 分配新的宿主机端口（或重用原端口）
├─ 启动 OpenCode Serve
├─ 更新 Redis 沙箱元数据
│   └─ status = "running", 新的 PID, port
└─ 请求正常处理

步骤 3: 管理员干预（极端情况）
├─ 场景1: 自动恢复失败
│   ├─ 多次重建仍然失败
│   ├─ POST /admin/sandbox/get (body: `{"domainID": "xxx", "sandboxID": "xxx"}`) 查看详情和日志
│   ├─ 分析根因（端口冲突、资源不足等）
│   └─ POST /admin/sandbox/destroy (body: `{"domainID": "xxx", "sandboxID": "xxx"}`) 清理残留
│   └─ 手动解决问题后等待下次请求触发创建
├─ 场景2: 需要保留现场排查
│   ├─ POST /admin/sandbox/get (body: `{"domainID": "xxx", "sandboxID": "xxx"}`) 导出日志和状态
│   ├─ 分析问题原因
│   └─ POST /admin/sandbox/create (body: `{"domainID": "xxx", "sandboxID": "xxx"}`) 重新创建
└─ 场景3: 资源泄漏或性能问题
    ├─ POST /admin/sandbox/list 查看所有沙箱状态
    ├─ POST /admin/sandbox/destroy (body: `{"domainID": "xxx", "sandboxID": "xxx"}`) 清理异常沙箱
    └─ 调整配置或扩容资源
```

**关键点**:
- **自动恢复优先**: 检测到故障立即重建，保证服务可用性
- **数据不丢失**: NAS 数据持久化，重建后自动恢复
- **状态一致性**: 重建过程中更新 Redis 状态，避免路由错误
- **可观测性**: 记录故障日志，支持事后分析
- **人工兜底**: 极端情况下通过管理接口人工介入

**重建期间的请求处理**:
- Engine 使用 FastAPI 框架（或类似异步框架），每个请求独立线程处理
- 重建过程中新到达的请求在请求队列中排队积压
- 请求线程同步等待沙箱重建完成，重试连接沙箱
- **重试策略**: 重试 3 次，每次等待自动维护线程修复沙箱
- **超时处理**: 重试 3 次仍失败，返回错误信息给上游，说明沙箱不可用原因

### 4.4 Echo-Agentic-Engine 处理流程（伪代码）

```
func handleRequest(req *http.Request) *http.Response {
    // 1. 提取请求标识
    domainID := req.Header.Get("X-Domain-ID")
    sandboxID := req.Header.Get("X-Sandbox-ID")

    // 2. 检查本地沙箱
    sandbox := localSandboxes.Get(domainID, sandboxID)

    if sandbox == nil {
        // 2.1 创建并启动沙箱
        sandbox = createAndStartSandbox(domainID, sandboxID)
        // 2.2 启动 OpenCode Serve (监听宿主机端口)
        sandbox.StartOpenCode()
        // 2.3 启动空闲计时器
        sandbox.StartIdleTimer(idleTimeout, func() {
            // 超时回调：销毁沙箱
            destroySandbox(sandbox)
        })
        // 2.4 注册到 Redis
        now := time.Now().Format(time.RFC3339)
        redis.Set("sandbox:"+domainID+":"+sandboxID, {
            "engine_ip": localIP,
            "engine_port": 8080,
            "sandbox_port": sandbox.HostPort,  // 宿主机映射端口 (如 30001)
            "status": "running",
            "pid": sandbox.PID,
            "created_at": now,
            "last_active_at": now,
        })
    }

    // 3. 更新活跃时间
    now := time.Now().Format(time.RFC3339)
    redis.HSet("sandbox:"+domainID+":"+sandboxID, "last_active_at", now)
    sandbox.ResetIdleTimer()  // 重置空闲计时器

    // 4. 路径转换：去掉 /trans 前缀
    originalPath := req.URL.Path
    targetPath := strings.TrimPrefix(originalPath, "/trans")

    // 5. 构建转发请求
    forwardReq := req.Clone(context.Background())
    forwardReq.URL.Path = targetPath
    forwardReq.URL.RawPath = targetPath
    forwardReq.Header.Del("X-Domain-ID")    // 移除标识 header
    forwardReq.Header.Del("X-Sandbox-ID")

    // 6. 透传请求到沙箱内的 OpenCode Serve
    // 注意: 沙箱内 OpenCode Serve 监听 127.0.0.1:4096
    // Bubblewrap 端口映射: 宿主机端口 → 沙箱内 4096
    targetURL := fmt.Sprintf("http://127.0.0.1:%d%s", sandbox.HostPort, targetPath)
    return httpProxy.Forward(forwardReq, targetURL)
}

func destroySandbox(sandbox *Sandbox) {
    // 1. 标记销毁中
    redis.HSet("sandbox:"+sandbox.DomainID+":"+sandbox.SandboxID, "status", "destroying")

    // 2. 杀掉进程
    sandbox.Process.Kill()

    // 3. 清理资源
    sandbox.Cleanup()

    // 4. 删除 Redis 记录
    redis.Del("sandbox:" + sandbox.DomainID + ":" + sandbox.SandboxID)

    // 5. 从本地列表移除
    localSandboxes.Delete(sandbox.DomainID, sandbox.SandboxID)
}
```

## 8. 安全机制

### 8.1 白名单控制
- **职责归属**: 白名单验证由上游网关 (Echo-Agent) 负责，不在本系统范围内
- **本系统职责**: 仅根据 domainID 进行存储隔离和沙箱管理
- **配置**: domainID 白名单由上游维护到 redis（具体机制不关心）

### 8.2 沙箱隔离

**Bubblewrap 隔离机制**:
- **PID namespace**: 进程树隔离,看不到宿主进程
- **Mount namespace**: 文件系统隔离,独立挂载视图
- **IPC namespace**: 进程间通信隔离
- **UTS namespace**: 主机名隔离
- **User namespace** (可选): UID/GID 映射,非特权用户运行

**安全特性**:
- ✅ 最小权限原则: 只挂载必需目录
- ✅ 只读挂载: 系统目录使用 `--ro-bind`
- ✅ 进程隔离: `--unshare-pid` 防止进程逃逸
- ✅ 自动清理: `--die-with-parent` 防止孤儿进程
- ✅ 会话隔离: `--new-session` 防止终端劫持

**与 Docker 对比**:

| 特性 | Bubblewrap | Docker |
|------|-----------|--------|
| 攻击面 | 极小（几千行代码） | 较大（完整运行时） |
| 守护进程 | 无 | 有（dockerd） |
| 特权要求 | 非特权用户 | 需要 root/docker 组 |
| 配置复杂度 | 高（手动配置） | 低（预设模板） |
| 安全性 | 默认安全 | 依赖配置 |

### 8.3 文件系统安全

**隔离策略**:

沙箱内可见:
- `/sandbox/defaultProject/` - 读写挂载（NAS）
- `/usr /lib /lib64 /bin` - 只读挂载（系统）
- `/tmp /var` - tmpfs（临时）
- `/proc /dev` - 虚拟文件系统

沙箱内不可见:
- 宿主机的其他目录
- 其他沙箱的 NAS 目录

**NAS 隔离**:
- 按 domainID 隔离存储空间
- 不同 domainID 的沙箱无法互相访问
- 同一 domainID 下的不同 sandboxID 也相互隔离

### 8.4 网络安全

**当前网络策略**:
- **模式**: 隔离网络命名空间（使用 `--unshare-net`）
- **端口映射**: 使用 `--port {宿主机端口}:4096` 将沙箱内端口映射到宿主机
- **原因**:
  - 提升安全性，沙箱内进程无法直接访问外部网络
  - 通过端口映射精确控制访问
  - 避免沙箱间网络干扰

**网络访问方案**:
- **AI API 调用**: 沙箱内 OpenCode Serve 需要访问外部 AI API
- **解决方案**:
  - 配置网络转发或代理（如 slirp4netns）
  - 或使用宿主机网络代理
  - 内网环境已具有网络通路

**安全优势**:
- 网络隔离，沙箱间互不影响
- 沙箱无法直接访问外部网络，降低风险
- 通过端口映射精确控制入站流量

### 9.2 故障场景与恢复策略

| 故障类型 | 检测方式 | 恢复策略 | RTO |
|---------|---------|---------|-----|
| 沙箱进程崩溃 | 进程监控 | 自动重建, NAS 恢复 | < 10s |
| Engine 节点故障 | 心跳超时 | 流量切换其他节点 | < 30s |
| Redis 故障 | 连接超时 | 主从切换/哨兵 | < 60s |
| NAS 故障 | 挂载检测 | 多路径/冗余 | 视存储方案 |

**Redis 故障降级策略**:
- Redis 故障期间，Engine 直接拒绝所有请求
- 返回具体错误信息，说明 Redis 连接问题
- 不使用本地缓存，避免同步复杂性和数据不一致
- Redis 性能够快，无需本地缓存优化

### 9.3 故障恢复
- **沙箱故障**: 自动重建,从 NAS 恢复 workspace
- **Engine 节点故障**: 上游网关检测心跳超时,流量切换到其他节点
- **跨节点恢复**: 新节点可基于 NAS 数据恢复沙箱状态

### 9.4 无状态设计
- Echo-Agentic-Engine 节点不持久化状态,全部存储到 redis + NAS
- 支持水平扩展,节点可随时增删

## 11. 常见问题与约束

### 11.1 FAQ（高优先级）

**OpenCode Serve API**:
- ✅ **Session ID 管理**: 不需要 Engine 管理 session 恢复。Engine 只负责拉起 OpenCode Serve 并暴露 API 接口，由 Echo-Agent 通过 API 选择会话、触发 resume 等操作。
- ✅ **启动命令**: `opencode serve --hostname=127.0.0.1 --port=4096`（已确认，监听沙箱内 127.0.0.1:4096）
- ✅ **API 接口定义**: Engine 不需要定义接口，只透传 OpenCode Serve 的原生接口（包括 HTTP 和流式接口）。Engine 职责：提取 domainID/sandboxID header 后透传请求。
- ✅ **Project 切换**: 取决于 OpenCode Serve 是否支持，Engine 只做透传，无需讨论。
- ✅ **配置文件格式**: 大模型配置在 `opencode.json` 中，遵循 OpenCode 格式协议。由文件管理系统负责，用户通过 Web UI 修改工程级或用户级 `opencode.json`。提供默认配置，考虑模型接口不暴露的原则，Web UI 可能会隐藏 `opencode.json`。
- ✅ **健康检测接口**: `GET /global/health`，响应格式：`{"healthy": true, "version": "string"}`

**白名单更新机制**:
- ✅ **机制**: Web UI 配置时更新到 Redis，Engine 每次收到请求时查询 Redis 验证。
- **延迟容忍度**: 实时查询，无延迟问题。

**网络隔离策略**:
- ✅ **联网需求**: OpenCode Serve 需要联网（调用 AI API），内网环境部署，具有网络通路。
- ✅ **网络模式**: 使用 `--unshare-net` 隔离网络命名空间，配合端口映射 `--port {宿主机端口}:4096`。
- ✅ **网络访问**: 通过配置网络转发或代理实现 AI API 访问。

### 11.2 FAQ（中优先级）

**空闲超时配置**:
- ✅ **默认时长**:
  - Bubblewrap 沙箱空闲默认时长：1小时
  - Echo-Agent 与 Engine 心跳超时时长：30秒
  - Engine 与沙箱心跳超时时长：30秒
- ✅ **动态调整**: 支持在 Web UI 配置所有超时时长，Echo-Agent 设置到 Redis，Engine 读取使用
- ✅ **持久化**: 不需要，沙箱状态由 OpenCode 自行维护在 NAS 中

**Bubblewrap 配置**:
- ✅ **资源限制**: 暂不需要限制 CPU/内存（可选优化项，可配合 cgroups）
- ✅ **并发限制**: 单 Engine 默认最大 50 个沙箱，可在 Web UI 配置，Echo-Agent 设置到 Redis，Engine 读取
- ✅ **并发处理**: Echo-Agent 负责请求排队和流量调度。若 Engine 沙箱数达上限，Echo-Agent 将请求排队或调度到其他 Engine 节点。无需 Engine 自己实现排队机制
- ✅ **用户权限**: 使用默认配置即可，外部通过 OpenCode Serve 接口使用沙箱，不需要获取沙箱用户权限

**故障恢复细节**:
- ✅ **崩溃检测**: Engine 轮询发送 `GET /global/health` 请求给沙箱内的 OpenCode Serve，响应格式：`{"healthy": true, "version": "string"}`，超时时长 30秒
- ✅ **日志保存**: 保存到映射给沙箱的 NAS 路径中，按照 OpenCode 默认 log 位置即可（沙箱内是独立环境）
- ✅ **Session 数据**: OpenCode 自行维护和持久化到 NAS，Engine 不需要管理

### 11.3 系统约束

**OpenCode 约束**:
- 单沙箱单 project，不支持多 project
- Workspace 目录必须为 `/defaultProject/`
- Project 名固定为 `default`，不需要传入

**Bubblewrap 约束**:
- 需要 Linux 3.8+ 内核支持 user namespace
- 非 privileged 用户运行需要 `--unshare-user`
- 网络功能有限，需要外部工具配置复杂网络

**存储约束**:
- NAS 必须在所有 Echo-Agentic-Engine 节点上挂载到相同路径
- Redis 需要高可用配置
- 文件操作独立于沙箱运行状态

### 11.4 技术风险

**资源管理（中风险）**:
- Bubblewrap 默认无资源限制
- 恶意沙箱可能耗尽宿主机资源
- 需要配合 systemd/cgroups 实现资源限制（可选）

## 16. 已明确的技术细节总结

### 16.1 端口映射机制

**访问链路**:
```
Echo-Agent → Engine:{宿主机IP}:8080 → 127.0.0.1:{sandbox_port}(30001-40000) → Bubblewrap端口映射 → 沙箱内 127.0.0.1:4096
```

**端口池管理**:
- 端口范围: 30001-40000
- 分配策略: 随机分配
- 冲突避免: 分配前查询 Redis 获取本机所有沙箱端口
- 回收策略: 沙箱销毁后立即回收

### 16.2 网络访问方案

**网络模式**:
- 使用 `--unshare-net` 隔离网络命名空间
- 通过 `--port {宿主机端口}:4096` 进行端口映射（类似 Docker `-p` 参数）
- 内网环境已具有网络通路，OpenCode Serve 可访问外部 AI API

### 16.3 健康检测机制

**检测接口**:
- 方法: `GET /global/health`
- 响应: `{"healthy": true, "version": "string"}`
- 轮询频率: 30秒超时

### 16.4 并发与流量控制

**并发处理**:
- Engine 不实现排队机制，由 Echo-Agent 负责请求排队和流量调度
- 单 Engine 默认最大 50 个沙箱（可配置）
- 达到上限后 Echo-Agent 将请求调度到其他 Engine 节点

### 16.5 故障降级策略

**Redis 故障处理**:
- Redis 故障期间 Engine 直接拒绝所有请求
- 返回具体错误信息说明 Redis 连接问题
- 不使用本地缓存，避免同步复杂性

**沙箱重建期间请求处理**:
- 使用 FastAPI 框架，每个请求独立线程处理
- 重建过程中新请求在队列排队积压
- 重试策略: 重试 3 次，等待维护线程修复沙箱
- 超时处理: 重试 3 次仍失败则返回错误

### 16.6 配置更新生效

**生效策略**:
- `max_sandboxes_per_engine` 更新: 立即生效，超过新上限的多余沙箱直接销毁
- `sandbox_idle_timeout` 更新: 立即生效，已运行沙箱的空闲计时器立即使用新值
- 其他超时配置: 立即生效，适用于所有沙箱

### 16.7 管理接口认证

**认证机制**:
- 所有管理接口不需要认证
- 可直接访问使用

### 16.8 NAS 目录创建

**创建规则**:
- 以 root 身份创建
- 权限: 可读可写（保证脚本如 Python 可执行）
- 失败处理: 返回错误信息并记录日志

### 16.9 大模型配置

**配置方式**:
- 大模型配置在 `opencode.json` 中，遵循 OpenCode 格式协议
- 由文件管理系统负责维护
- 不通过环境变量注入
