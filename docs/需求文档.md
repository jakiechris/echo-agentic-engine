## 1. 整体架构

### 1.1 系统全景

```
用户界面层
├─ OpenCode UI (聊天交互界面)
└─ 文件管理 Web UI

服务网关层
└─ Echo-Agent (请求路由 + 文件服务)

沙箱管理层
└─ Echo-Agentic-Engine (本系统)

沙箱运行层
└─ Bubblewrap Sandbox → OpenCode Serve

存储层
├─ Redis (状态 + 白名单 + 节点注册)
└─ NAS (持久化存储)
```

### 1.2 核心组件职责

| 组件 | 职责 | 边界 |
|-----|------|------|
| **OpenCode UI** | 用户聊天交互界面 | 不感知沙箱,仅通过 echo-agent 通信 |
| **Echo-Agent** | 请求路由、负载均衡、文件管理、白名单管理 | 不直接管理沙箱,只做转发 |
| **Echo-Agentic-Engine** | 沙箱生命周期管理、请求透传 | 轻量化,不处理业务逻辑 |
| **OpenCode Serve** | AI 对话服务、session 管理 | 单实例单 project,原生 API |
| **文件 Web UI** | 文件上传/下载 | 由 echo-agent 独立实现 |

## 2. Echo-Agentic-Engine 核心功能

### 2.1 请求路由
- 每个请求通过 `domainID` + `sandboxID` 唯一标识沙箱
- 从 HTTP Headers 提取 `X-Domain-ID` 和 `X-Sandbox-ID`
- 白名单验证由上游网关负责，本系统默认 domainID 合法

### 2.2 反向代理

Echo-Agentic-Engine 对外暴露统一代理路径，所有请求通过 header 携带标识信息：

**对外接口**：
```
ANY  /trans/{path}                   # 统一代理路径（path 为 OpenCode Serve 原生路径）
```

**请求标识**（HTTP Headers）：
- `X-Domain-ID`: 租户标识（用于白名单验证和存储隔离）
- `X-Sandbox-ID`: 沙箱标识（用于定位具体沙箱实例）

**路由规则**：
```
客户端请求:  POST /trans/api/chat
Headers:     X-Domain-ID: tenant-001
            X-Sandbox-ID: sandbox-abc
            (其他原有headers)

转发目标:    POST http://127.0.0.1:{port}/api/chat
Headers:     (保留所有原始headers，移除 X-Domain-ID 和 X-Sandbox-ID)
```

**支持的所有 OpenCode Serve 原生接口**：
- HTTP REST API：对话、文件操作、配置管理等
- SSE 事件流：`GET /trans/api/event` → `GET /api/event`
- WebSocket 连接：`WS /trans/api/pty/{id}/connect` → `WS /api/pty/{id}/connect`

**关键特性**：
- ✅ 路径透明映射：去掉 `/trans` 前缀后原样透传
- ✅ Header 处理：提取 `domainID`/`sandboxID` 用于路由，不传递给 OpenCode
- ✅ 协议无关：支持 HTTP、SSE、WebSocket 等所有协议
- ✅ 无业务逻辑：不修改请求/响应内容，纯透传

### 2.3 沙箱管理

**生命周期管理职责**:
- Engine 完全负责沙箱的创建、监控、销毁
- 上游网关（Echo-Agent）只负责路由，不感知沙箱生命周期
- 沙箱自动销毁机制：长时间无请求自动清理

**创建沙箱**
```bash
bwrap \
  # ── 系统目录只读 ──
  --ro-bind /usr /usr \
  --ro-bind /etc/resolv.conf /etc/resolv.conf \
  --ro-bind /etc/ssl /etc/ssl \
  --ro-bind /lib /lib \          # 条件挂载
  --ro-bind /lib64 /lib64 \      # 条件挂载
  --ro-bind /bin /bin \           # 条件挂载
  --ro-bind /sbin /sbin \         # 条件挂载
  --ro-bind /proc /proc \
  --dev /dev \
  #
  # ── NAS 数据挂载（关键映射）──
  --bind /nas/{domainID}/{sandboxID} /sandbox \
  --bind /nas/{domainID}/{sandboxID}/defaultProject /sandbox/defaultProject \
  #
  # ── 命名空间隔离 ──
  --unshare-user \
  --unshare-pid \
  --die-with-parent \
  --new-session \
  #
  # ── 工作目录与环境变量 ──
  --chdir /sandbox/defaultProject \
  --setenv PATH "/usr/local/bin:/usr/bin:/bin" \
  --setenv HOME "/tmp" \
  --setenv XDG_DATA_HOME "/sandbox/data" \
  --setenv XDG_CONFIG_HOME "/sandbox/config" \
  --setenv XDG_STATE_HOME "/sandbox/data/state" \
  --setenv XDG_CACHE_HOME "/sandbox/data/cache" \
  --setenv OPENCODE_SERVER_USERNAME "opencode" \
  --setenv OPENCODE_SERVER_PASSWORD "{随机 UUID}" \
  --setenv OPENCODE_CONFIG_CONTENT '{大模型配置 JSON}' \
  #
  # ── 启动 opencode ──
  -- opencode serve \
  --hostname=127.0.0.1 \
  --port=4096    # 沙箱内固定监听 4096 端口
```

**端口映射说明**:
- OpenCode Serve 在沙箱内固定监听 `127.0.0.1:4096`
- Bubblewrap 本身不提供端口映射功能
- Echo-Agentic-Engine 作为反向代理实现端口转发:
  ```
  外部请求 → Engine:{宿主机端口}
           → Engine 内部路由 (根据 sandboxID)
           → 沙箱内 127.0.0.1:4096
  ```
- 示例:
  - 沙箱1: Engine 监听 30001 → 转发到沙箱内 127.0.0.1:4096
  - 沙箱2: Engine 监听 30002 → 转发到沙箱内 127.0.0.1:4096
- Redis 记录 `sandbox_port` 字段存储宿主机映射端口（30001, 30002...）

关键参数：

| 参数 | 作用 |
|------|------|
| `--ro-bind src dst` | 只读挂载，沙箱内无法修改系统文件 |
| `--bind src dst` | 可读写挂载，用户数据目录 |
| `--dev /dev` | 仅生成 null/zero/full/random/urandom/tty |
| `--unshare-user` | 用户命名空间，无需 root |
| `--unshare-pid` | PID 命名空间，看不到其他进程 |
| `--die-with-parent` | Gateway 退出时内核自动 SIGKILL 子进程 |
| `--new-session` | 独立进程组，信号不跨沙箱传播 |
| `OPENCODE_CONFIG_CONTENT` | 注入大模型 provider 配置（API 地址、密钥） |

**关键参数说明**:
- `--bind /nas/{domainID}/{sandboxID} /sandbox`: NAS 存储映射到沙箱根目录
- `--bind` vs `--ro-bind`: NAS 使用读写挂载，系统目录只读
- `--die-with-parent`: Echo-Agentic-Engine 崩溃时自动清理沙箱
- `--new-session`: 安全隔离，防止终端劫持
- `--port 4096`: OpenCode Serve 在沙箱内固定监听 4096 端口
- 网络: 暂不隔离（`--unshare-net` 待确认 opencode 是否需要联网）

**生命周期管理**:
- **创建**:
  - Engine 收到请求时检查本地沙箱是否存在
  - 不存在则创建并启动 OpenCode Serve
  - 创建成功后写入 Redis: `sandbox:{domainID}:{sandboxID}`
- **运行**:
  - 每次收到该沙箱的请求，重置空闲计时器
  - 定时向 Redis 更新心跳状态
- **自动销毁**:
  - 定时器监控沙箱空闲时长（默认配置，如 30 分钟无请求）
  - 超时前写入 Redis 标记 `status: "destroying"`
  - 销毁沙箱进程，清理资源
  - 销毁完成后从 Redis 删除沙箱记录
- **管理接口**: 支持手动停止/启动/查询沙箱

### 2.4 空闲计时器机制

**设计目标**:
- 自动回收长期无请求的沙箱，释放资源
- 避免沙箱无限累积导致资源耗尽
- 支持用户长时间交互后的会话恢复

**实现机制**:
```
每个沙箱维护独立的计时器:
┌─────────────────────────────────────┐
│  Sandbox Instance                    │
│                                      │
│  • idleTimer: Timer                  │
│  • idleTimeout: 30min (可配置)        │
│  • lastActiveAt: timestamp           │
│                                      │
│  ResetIdleTimer():                   │
│    - 停止旧计时器                     │
│    - 启动新计时器(idleTimeout)        │
│    - 更新 lastActiveAt               │
│                                      │
│  OnTimeout():                        │
│    - 标记 Redis: status="destroying" │
│    - 销毁沙箱                        │
│    - 删除 Redis 记录                 │
└─────────────────────────────────────┘
```

**关键流程**:
1. **沙箱创建时**: 启动空闲计时器，设置超时回调
2. **请求到达时**: 调用 `ResetIdleTimer()` 重置计时器
3. **超时触发时**:
   - 标记 Redis: `status = "destroying"`
   - 杀掉沙箱进程
   - 清理资源（端口回收等）
   - 删除 Redis 记录

**配置参数**:
- `idleTimeout`: 空闲超时时长（默认 30 分钟，可配置）
- `timerCheckInterval`: 计时器检查间隔（可选，用于集中管理方案）

**注意事项**:
- 每个沙箱独立计时器，互不影响
- 计时器精度要求不高，允许 ±1 分钟误差
- 销毁操作需要考虑优雅关闭（可选）

### 2.5 状态同步

**同步时机**:
| 事件 | 同步内容 | 目标 |
|------|---------|------|
| 沙箱创建 | 沙箱信息 + 本机 IP + 创建时间 | Redis |
| 沙箱请求 | 更新最后活跃时间 | Redis |
| 沙箱空闲超时 | 标记 `status: "destroying"` | Redis |
| 沙箱销毁完成 | 删除沙箱记录 | Redis |
| 心跳上报 | 节点负载 + 时间戳 | Redis |

**Redis 数据结构**:
```
# 沙箱元数据 (Hash)
sandbox:{domainID}:{sandboxID} = {
    "engine_ip": "x.x.x.x",
    "engine_port": 8080,
    "sandbox_port": 30001,
    "status": "running",          # running / destroying
    "pid": 12345,
    "created_at": "2026-03-13T10:00:00Z",
    "last_active_at": "2026-03-13T10:30:00Z"  # 最后活跃时间
}

# 关键说明:
# - engine_port: Echo-Agentic-Engine 对外服务端口（固定 8080）
# - sandbox_port: 沙箱在宿主机的映射端口（动态分配，如 30001, 30002...）
# - 沙箱内 OpenCode Serve 固定监听 4096 端口
# - Bubblewrap 启动时需配置端口映射: 宿主机端口 → 沙箱内 4096
# - last_active_at: 每次请求更新，用于上游网关判断沙箱存活

# 节点信息 (Hash)
engine:nodes = {
    "10.0.1.1:8080": {"capacity": 500, "running": 123, "heartbeat": 1709520060},
    "10.0.1.2:8080": {"capacity": 500, "running": 89, "heartbeat": 1709520061}
}

# 白名单 (Set)
whitelist:domainIDs = ["domain1", "domain2"]
```

**心跳机制**:
```
Echo-Agentic-Engine 节点 ──(每 10s)──→ Redis 更新 heartbeat

上游网关检查逻辑:
if (now - heartbeat) > 30s:
    标记节点离线
    从可用列表移除
```
- 沙箱创建后,将信息 + 本机 IP 写入 redis
- 沙箱存活状态实时同步
- Session 信息同步（待实现）

## 3. 存储架构

### 3.1 NAS 共享存储

**挂载说明**:
- NAS 由后台 K8s 自动提供挂载初始化
- 根路径固定为 `/nas`，无需手动挂载
- 所有 Echo-Agentic-Engine 节点共享同一 NAS 存储

**跨节点共享**:
- 多 Echo-Agentic-Engine 节点共享同一 NAS
- 支持沙箱跨节点恢复
- 文件操作独立于沙箱运行状态

### 3.2 目录结构

**沙箱根目录**:
```
/nas/{domainID}/{sandboxID}/       # 沙箱根目录
  └── defaultProject/              # opencode workspace (必须此名称)
      ├── .config/
      │   └── .opencode/
      │       └── opencode.json    # 工程级配置
      ├── .sessions/               # session 数据（待确认）
      └── [用户文件]                # workspace 文件
```

**从沙箱内视角**:
```
/sandbox/                          # 沙箱根挂载点
  └── defaultProject/              # opencode workspace
      ├── .config/.opencode/opencode.json
      └── [用户文件]
```

**挂载映射说明**:
- 宿主机路径: `/nas/{domainID}/{sandboxID}/`
- 沙箱内路径: `/sandbox/`
- Bubblewrap 启动时通过 `--bind` 参数映射:
  ```bash
  bwrap --bind /nas/{domainID}/{sandboxID} /sandbox ...
  ```

**OpenCode 启动**:
```bash
cd /sandbox/defaultProject
opencode serve  # 启动 OpenCode Server
```

### 3.3 配置文件

**opencode.json**:
- 位置: `/defaultProject/.config/.opencode/opencode.json`
- 作用: 工程级配置（API keys、模型参数等）
- 格式: 待查阅 OpenCode 官方文档

### 3.4 数据持久化

**NAS 存储**:
- Workspace 文件
- Session 数据（依赖 OpenCode 实现）
- 配置文件

**约定**:
- 单沙箱对应单 project(default),不支持多 project
- session 由 opencode API 原生管理
- 文件路径固定为 `/defaultProject/`

**Echo-Agentic-Engine 输入规范**:
- 请求来源: Echo-Agent（上游网关，已实现，具体实现不关心）
- 请求格式:
  ```
  ANY /trans/{path}
  Headers:
    - X-Domain-ID: {domainID}      # 租户标识
    - X-Sandbox-ID: {sandboxID}    # 沙箱标识
    - [其他 OpenCode 原生 headers] # 完整透传
  Body: OpenCode 原生请求体
  ```

**上游路由机制说明**（仅供参考，不关心具体实现）:
- 沙箱存活: 查询 Redis 发现 `sandbox:{domainID}:{sandboxID}` 存在 → 直接转发到对应 Engine
- 沙箱不存在: Redis 中无该沙箱记录 → 选择负载最低的 Engine（`engine:nodes` 中 running/capacity 最小）→ 转发请求
- Engine 职责: 收到请求后自行判断是否创建沙箱，并管理沙箱生命周期
- Engine 离线: 心跳超时时由上游从 `engine:nodes` 移除节点，不参与路由

### 4.4 Echo-Agentic-Engine 处理流程（伪代码）

```
func handleRequest(req *http.Request) *http.Response {
    // 1. 提取请求标识
    domainID := req.Header.Get("X-Domain-ID")
    sandboxID := req.Header.Get("X-Sandbox-ID")

    // 2. 检查本地沙箱
    sandbox := localSandboxes.Get(domainID, sandboxID)

    if sandbox == nil {
        // 2.1 创建沙箱
        sandbox = createSandbox(domainID, sandboxID)
        // 2.2 启动 OpenCode Serve (监听沙箱内 4096 端口)
        sandbox.StartOpenCode()
        // 2.3 启动空闲计时器
        sandbox.StartIdleTimer(idleTimeout, func() {
            // 超时回调：销毁沙箱
            destroySandbox(sandbox)
        })
        // 2.4 注册到 Redis
        now := time.Now().Format(time.RFC3339)
        redis.Set("sandbox:"+domainID+":"+sandboxID, {
            "engine_ip": localIP,
            "engine_port": 8080,
            "sandbox_port": sandbox.HostPort,  // 宿主机映射端口 (如 30001)
            "status": "running",
            "pid": sandbox.PID,
            "created_at": now,
            "last_active_at": now,
        })
    }

    // 3. 更新活跃时间
    now := time.Now().Format(time.RFC3339)
    redis.HSet("sandbox:"+domainID+":"+sandboxID, "last_active_at", now)
    sandbox.ResetIdleTimer()  // 重置空闲计时器

    // 4. 路径转换：去掉 /trans 前缀
    originalPath := req.URL.Path
    targetPath := strings.TrimPrefix(originalPath, "/trans")

    // 5. 构建转发请求
    forwardReq := req.Clone(context.Background())
    forwardReq.URL.Path = targetPath
    forwardReq.URL.RawPath = targetPath
    forwardReq.Header.Del("X-Domain-ID")    // 移除标识 header
    forwardReq.Header.Del("X-Sandbox-ID")

    // 6. 透传请求到沙箱内的 OpenCode Serve
    // 注意: 沙箱内 OpenCode Serve 固定监听 127.0.0.1:4096
    // sandbox.HostPort 是宿主机映射端口，Engine 通过内部路由转发到对应沙箱的 4096 端口
    targetURL := fmt.Sprintf("http://127.0.0.1:4096%s", targetPath)
    return httpProxy.Forward(forwardReq, targetURL)
}

func destroySandbox(sandbox *Sandbox) {
    // 1. 标记销毁中
    redis.HSet("sandbox:"+sandbox.DomainID+":"+sandbox.SandboxID, "status", "destroying")

    // 2. 杀掉进程
    sandbox.Process.Kill()

    // 3. 清理资源
    sandbox.Cleanup()

    // 4. 删除 Redis 记录
    redis.Del("sandbox:" + sandbox.DomainID + ":" + sandbox.SandboxID)

    // 5. 从本地列表移除
    localSandboxes.Delete(sandbox.DomainID, sandbox.SandboxID)
}
```

## 7. 设计原则

- **轻量**: 纯透传 opencode serve 接口,最小化额外逻辑
- **无状态**: 依赖 redis 和 NAS,Echo-Agentic-Engine 节点可水平扩展
- **高可用**: 多节点共享存储,支持故障恢复
- **自管理**: 沙箱生命周期完全由 Engine 管理，包括自动销毁机制

## 8. 安全机制

### 8.1 白名单控制
- **职责归属**: 白名单验证由上游网关 (Echo-Agent) 负责，不在本系统范围内
- **本系统职责**: 仅根据 domainID 进行存储隔离和沙箱管理
- **配置**: domainID 白名单由上游维护到 redis（具体机制不关心）

### 8.2 沙箱隔离

**Bubblewrap 隔离机制**:
- **PID namespace**: 进程树隔离,看不到宿主进程
- **Mount namespace**: 文件系统隔离,独立挂载视图
- **IPC namespace**: 进程间通信隔离
- **UTS namespace**: 主机名隔离
- **User namespace** (可选): UID/GID 映射,非特权用户运行

**安全特性**:
- ✅ 最小权限原则: 只挂载必需目录
- ✅ 只读挂载: 系统目录使用 `--ro-bind`
- ✅ 进程隔离: `--unshare-pid` 防止进程逃逸
- ✅ 自动清理: `--die-with-parent` 防止孤儿进程
- ✅ 会话隔离: `--new-session` 防止终端劫持

**与 Docker 对比**:

| 特性 | Bubblewrap | Docker |
|------|-----------|--------|
| 攻击面 | 极小（几千行代码） | 较大（完整运行时） |
| 守护进程 | 无 | 有（dockerd） |
| 特权要求 | 非特权用户 | 需要 root/docker 组 |
| 配置复杂度 | 高（手动配置） | 低（预设模板） |
| 安全性 | 默认安全 | 依赖配置 |

### 8.3 文件系统安全

**隔离策略**:

沙箱内可见:
- `/sandbox/defaultProject/` - 读写挂载（NAS）
- `/usr /lib /lib64 /bin` - 只读挂载（系统）
- `/tmp /var` - tmpfs（临时）
- `/proc /dev` - 虚拟文件系统

沙箱内不可见:
- 宿主机的其他目录
- 其他沙箱的 NAS 目录

**NAS 隔离**:
- 按 domainID 隔离存储空间
- 不同 domainID 的沙箱无法互相访问
- 同一 domainID 下的不同 sandboxID 也相互隔离

### 8.4 网络安全

**网络策略（待确认）**:
- 方案 1: `--unshare-net` 完全隔离网络（如果 opencode 不需要联网）
- 方案 2: `--share-net` 共享宿主网络（如果 opencode 需要联网）
- 方案 3: 配置网络命名空间 + 白名单（高级需求）

**风险**:
- 如果共享网络，沙箱内进程可以访问外部网络
- 可能需要配置防火墙规则限制网络访问

### 9.2 故障场景与恢复策略

| 故障类型 | 检测方式 | 恢复策略 | RTO |
|---------|---------|---------|-----|
| 沙箱进程崩溃 | 进程监控 | 自动重建, NAS 恢复 | < 10s |
| Engine 节点故障 | 心跳超时 | 流量切换其他节点 | < 30s |
| Redis 故障 | 连接超时 | 主从切换/哨兵 | < 60s |
| NAS 故障 | 挂载检测 | 多路径/冗余 | 视存储方案 |

### 9.3 故障恢复
- **沙箱故障**: 自动重建,从 NAS 恢复 workspace
- **Engine 节点故障**: 上游网关检测心跳超时,流量切换到其他节点
- **跨节点恢复**: 新节点可基于 NAS 数据恢复沙箱状态

### 9.4 无状态设计
- Echo-Agentic-Engine 节点不持久化状态,全部存储到 redis + NAS
- 支持水平扩展,节点可随时增删

### 11.3 系统约束

**OpenCode 约束**:
- 单沙箱单 project，不支持多 project
- Workspace 目录必须为 `/defaultProject/`
- Project 名固定为 `default`，不需要传入

**Bubblewrap 约束**:
- 需要 Linux 3.8+ 内核支持 user namespace
- 非 privileged 用户运行需要 `--unshare-user`
- 网络功能有限，需要外部工具配置复杂网络

**存储约束**:
- NAS 必须在所有 Echo-Agentic-Engine 节点上挂载到相同路径
- Redis 需要高可用配置
- 文件操作独立于沙箱运行状态

### 11.4 技术风险

**Session 管理（高风险）**:
- 如果 OpenCode API 不支持 sessionID，需要设计替代方案
- 可能需要拦截/修改 API 请求来注入 session 信息

**网络隔离（中风险）**:
- 如果 OpenCode 需要联网，安全隔离会受影响
- 可能需要配置网络白名单或代理

**资源管理（中风险）**:
- Bubblewrap 默认无资源限制
- 恶意沙箱可能耗尽宿主机资源
- 需要配合 systemd/cgroups 实现资源限制
