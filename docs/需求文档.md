## 1. 整体架构

### 1.1 系统全景

```
用户界面层
├─ OpenCode UI (聊天交互界面)
└─ 文件管理 Web UI

服务网关层
└─ Echo-Agent (请求路由 + 文件服务)

沙箱管理层
└─ Echo-Agentic-Engine (本系统)

沙箱运行层
└─ Bubblewrap Sandbox → OpenCode Serve

存储层
├─ Redis (状态 + 白名单 + 节点注册)
└─ NAS (持久化存储)
```

### 1.2 核心组件职责

| 组件 | 职责 | 边界 |
|-----|------|------|
| **OpenCode UI** | 用户聊天交互界面 | 不感知沙箱,仅通过 echo-agent 通信 |
| **Echo-Agent** | 请求路由、负载均衡、文件管理、白名单管理 | 不直接管理沙箱,只做转发 |
| **Echo-Agentic-Engine** | 沙箱生命周期管理、请求透传 | 轻量化,不处理业务逻辑 |
| **OpenCode Serve** | AI 对话服务、session 管理 | 单实例单 project,原生 API |
| **文件 Web UI** | 文件上传/下载 | 由 echo-agent 独立实现 |

## 2. Echo-Agentic-Engine 核心功能

### 2.1 请求路由
- 每个请求通过 `domainID` + `sandboxID` 唯一标识沙箱
- domainID 白名单验证(从 redis 获取,支持动态更新)
- 非法 domainID 拒绝请求

### 2.2 反向代理

Echo-Agentic-Engine 对外暴露统一代理路径，所有请求通过 header 携带标识信息：

**对外接口**：
```
ANY  /trans/{path}                   # 统一代理路径（path 为 OpenCode Serve 原生路径）
```

**请求标识**（HTTP Headers）：
- `X-Domain-ID`: 租户标识（用于白名单验证和存储隔离）
- `X-Sandbox-ID`: 沙箱标识（用于定位具体沙箱实例）

**路由规则**：
```
客户端请求:  POST /trans/api/chat
Headers:     X-Domain-ID: tenant-001
            X-Sandbox-ID: sandbox-abc
            (其他原有headers)

转发目标:    POST http://127.0.0.1:{port}/api/chat
Headers:     (保留所有原始headers，移除 X-Domain-ID 和 X-Sandbox-ID)
```

**支持的所有 OpenCode Serve 原生接口**：
- HTTP REST API：对话、文件操作、配置管理等
- SSE 事件流：`GET /trans/api/event` → `GET /api/event`
- WebSocket 连接：`WS /trans/api/pty/{id}/connect` → `WS /api/pty/{id}/connect`

**关键特性**：
- ✅ 路径透明映射：去掉 `/trans` 前缀后原样透传
- ✅ Header 处理：提取 `domainID`/`sandboxID` 用于路由，不传递给 OpenCode
- ✅ 协议无关：支持 HTTP、SSE、WebSocket 等所有协议
- ✅ 无业务逻辑：不修改请求/响应内容，纯透传

### 2.3 沙箱管理

**创建沙箱**
```bash
bwrap \
  # ── 系统目录只读 ──
  --ro-bind /usr /usr \
  --ro-bind /etc/resolv.conf /etc/resolv.conf \
  --ro-bind /etc/ssl /etc/ssl \
  --ro-bind /lib /lib \          # 条件挂载
  --ro-bind /lib64 /lib64 \      # 条件挂载
  --ro-bind /bin /bin \           # 条件挂载
  --ro-bind /sbin /sbin \         # 条件挂载
  --ro-bind /proc /proc \
  --dev /dev \
  #
  # ── 用户数据可读写 ──
  --bind /data/users/{scene}/{uid}/data       /data \
  --bind /data/users/{scene}/{uid}/config     /config \
  --bind /data/users/{scene}/{uid}/workspace  /workspace \
  --bind /data/users/{scene}/{uid}/tmp        /tmp \
  --ro-bind /data/users/{scene}/{uid}/workspace/.git  /workspace/.git \  # .git 只读保护（如存在）
  #
  # ── 命名空间隔离 ──
  --unshare-user \
  --unshare-pid \
  --die-with-parent \
  --new-session \
  #
  # ── 工作目录与环境变量 ──
  --chdir /workspace \
  --setenv PATH "/usr/local/bin:/usr/bin:/bin" \
  --setenv HOME "/tmp" \
  --setenv XDG_DATA_HOME "/data" \
  --setenv XDG_CONFIG_HOME "/config" \
  --setenv XDG_STATE_HOME "/data/state" \
  --setenv XDG_CACHE_HOME "/data/cache" \
  --setenv OPENCODE_SERVER_USERNAME "opencode" \
  --setenv OPENCODE_SERVER_PASSWORD "{随机 UUID}" \
  --setenv OPENCODE_CONFIG_CONTENT '{大模型配置 JSON}' \
  #
  # ── 启动 opencode ──
  -- opencode serve \
  --hostname=127.0.0.1 \
  --port={分配的端口}
```

关键参数：

| 参数 | 作用 |
|------|------|
| `--ro-bind src dst` | 只读挂载，沙箱内无法修改系统文件 |
| `--bind src dst` | 可读写挂载，用户数据目录 |
| `--dev /dev` | 仅生成 null/zero/full/random/urandom/tty |
| `--unshare-user` | 用户命名空间，无需 root |
| `--unshare-pid` | PID 命名空间，看不到其他进程 |
| `--die-with-parent` | Gateway 退出时内核自动 SIGKILL 子进程 |
| `--new-session` | 独立进程组，信号不跨沙箱传播 |
| `OPENCODE_CONFIG_CONTENT` | 注入大模型 provider 配置（API 地址、密钥） |

**关键参数说明**:
- `--bind` vs `--ro-bind`: NAS 使用读写挂载，系统目录只读
- `--die-with-parent`: Echo-Agentic-Engine 崩溃时自动清理沙箱
- `--new-session`: 安全隔离，防止终端劫持
- 网络: 暂不隔离（`--unshare-net` 待确认 opencode 是否需要联网）

**生命周期管理**:
- **创建**: 检查本地沙箱 → 不存在则创建 → 启动 opencode serve
- **查询**: 查询沙箱状态和位置信息
- **销毁**: 管理接口支持杀掉/启动沙箱
- **恢复**: ⚠️ **关键问题** - 需确认 session 恢复机制

### 2.4 Session 管理（关键问题）

**问题描述**:
OpenCode Serve 支持在一个 project 下管理多个 session，当前设计需要确认如何恢复特定 session。

```

│                                                         │
│  用户请求 ──→ 如何知道要恢复哪个 session?                │
│                                                         │
│  当前未知:                                              │

│  ├─ 响应中是否返回 sessionID?                           │
│  ├─ Session 创建/恢复/切换的 API 定义?                  │
│  └─ Session 持久化机制?                                 │

```

**确认项**:
| 项目 | 说明 | 解决方案 |
|------|------|--------|
| sessionID 传递方式 | API 请求/响应中如何携带 | 放到header中 |
| Session 创建 API | 如何创建新 session | 由上游echo-agent管理，通过opencode serve的api控制 |
| Session 恢复 API | 如何恢复已有 session | 由上游echo-agent管理，通过opencode serve的api控制 |
| Session 持久化 | Session 数据存储位置和格式 | 交给opencode，沙箱根目录会全部保存在nas |


### 2.5 状态同步

**同步时机**:
| 事件 | 同步内容 | 目标 |
|------|---------|------|
| 沙箱创建 | 沙箱信息 + 本机 IP | Redis |
| 沙箱销毁 | 删除沙箱记录 | Redis |
| 心跳上报 | 节点负载 + 时间戳 | Redis |
| Session 变更 | Session ID 映射 | Redis |

**Redis 数据结构**:
```
# 沙箱元数据 (Hash)
sandbox:{domainID}:{sandboxID} = {
    "engine_ip": "x.x.x.x",
    "engine_port": 8080,
    "sandbox_port": 30001,
    "status": "running",
    "pid": 12345,
    "created_at": "2026-03-13T10:00:00Z"
}

# 节点信息 (Hash)
gateway:nodes = {
    "10.0.1.1:8080": {"capacity": 500, "running": 123, "heartbeat": 1709520060},
    "10.0.1.2:8080": {"capacity": 500, "running": 89, "heartbeat": 1709520061}
}

# 白名单 (Set)
whitelist:domainIDs = ["domain1", "domain2"]
```

**心跳机制**:
```
Echo-Agentic-Engine 节点 ──(每 10s)──→ Redis 更新 heartbeat

Echo-Agent 检查逻辑:
if (now - heartbeat) > 30s:
    标记节点离线
    从可用列表移除
```
- 沙箱创建后,将信息 + 本机 IP 写入 redis
- 沙箱存活状态实时同步
- Session 信息同步（待实现）

## 3. 存储架构

### 3.1 NAS 共享存储

**挂载方式**:
```bash
# 宿主机挂载 NAS（CIFS/NFS/本地磁盘）
mount -t cifs //nas-server/share /mnt/nas -o user=username
# 或
mount -t nfs nas-server:/export/data /mnt/nas

# Bubblewrap 沙箱内访问
bwrap --bind /mnt/nas /nas ...
```

**跨节点共享**:
- 多 Echo-Agentic-Engine节点共享同一 NAS
- 支持沙箱跨节点恢复
- 文件操作独立于沙箱运行状态

### 3.2 目录结构

**沙箱根目录**:
```
/nas/{domainID}/{sandboxID}/       # 沙箱根目录
  └── defaultProject/              # opencode workspace (必须此名称)
      ├── .config/
      │   └── .opencode/
      │       └── opencode.json    # 工程级配置
      ├── .sessions/               # session 数据（待确认）
      └── [用户文件]                # workspace 文件
```

**从沙箱内视角**:
```
/sandbox/                          # 沙箱根挂载点
  └── defaultProject/              # opencode workspace
      ├── .config/.opencode/opencode.json
      └── [用户文件]
```

**OpenCode 启动**:
```bash
cd /sandbox/defaultProject
opencode serve start  # 具体命令待确认
```

### 3.3 配置文件

**opencode.json**:
- 位置: `/defaultProject/.config/.opencode/opencode.json`
- 作用: 工程级配置（API keys、模型参数等）
- 格式: 待查阅 OpenCode 官方文档

### 3.4 数据持久化

**NAS 存储**:
- Workspace 文件
- Session 数据（依赖 OpenCode 实现）
- 配置文件

**约定**:
- 单沙箱对应单 project(default),不支持多 project
- session 由 opencode API 原生管理
- 文件路径固定为 `/defaultProject/`

### 4.2 请求处理详细流程

#### 4.2.1 首次访问流程(沙箱不存在)

```
用户 → Echo-Agent → Echo-Agentic-Engine → OpenCode Serve

步骤 1: 用户发起请求
├─ HTTP Header: X-Domain-ID: {domainID}
├─ HTTP Header: X-Sandbox-ID: {sandboxID}
└─ 请求路径: /trans/{path}  (例如: /trans/api/chat)

步骤 2: Echo-Agent 网关处理
├─ 2.1 白名单验证
│   └─ 从 Redis 读取 whitelist:domainIDs
│   └─ 验证 domainID 是否在白名单中
│   └─ 非法则拒绝请求 (403 Forbidden)
├─ 2.2 查询沙箱位置
│   └─ Redis: GET sandbox:{domainID}:{sandboxID}
│   └─ 未找到 → 沙箱不存在
├─ 2.3 选择目标节点
│   └─ HGETALL gateway:nodes 获取所有节点
│   └─ 过滤: heartbeat > (now - 30s)
│   └─ 排序: running/capacity 升序
│   └─ 选择负载最低的节点
└─ 2.4 转发请求到目标 Echo-Agentic-Engine

步骤 3: Echo-Agentic-Engine 沙箱管理
├─ 3.1 本地沙箱检查
│   └─ 检查进程是否存在
│   └─ 检查端口是否监听
├─ 3.2 创建沙箱(不存在时)
│   └─ 准备 NAS 目录结构
│   └─ 生成随机密码 (UUID)
│   └─ 构建 Bubblewrap 命令
│   └─ 启动 OpenCode Serve
├─ 3.3 状态同步
│   └─ 写入 Redis: sandbox:{domainID}:{sandboxID}
│   └─ 更新节点负载: engine:{ip}:load
└─ 3.4 请求透传
    ├─ 提取 headers: X-Domain-ID, X-Sandbox-ID
    ├─ 路径转换: /trans/{path} → /{path}
    └─ 转发到 http://127.0.0.1:{port}/{path}

步骤 4: OpenCode Serve 处理
└─ 处理 AI 对话请求
└─ Session 创建/管理
```

#### 4.2.2 后续访问流程（沙箱已存在）

```
用户 → Echo-Agent → Echo-Agentic-Engine → OpenCode Serve

步骤 1-2: 同首次访问
├─ 白名单验证
├─ Redis 查询沙箱位置
└─ 找到沙箱 → 直接转发到对应 Echo-Agentic-Engine

步骤 3: Echo-Agentic-Engine 处理
├─ 3.1 本地沙箱检查
│   └─ 沙箱存在 → 直接转发
└─ 3.2 请求透传

步骤 4: OpenCode Serve 处理
└─ 处理请求
```

#### 4.2.3 故障恢复流程

```
检测到 Echo-Agentic-Engine 节点离线 (heartbeat 超时)

步骤 1: Echo-Agent 检测
├─ 节点 heartbeat > 30s 未更新
├─ 标记节点离线
└─ 从可用节点列表移除

步骤 2: 用户请求到达
├─ 沙箱记录指向离线节点
├─ 尝试转发 → 失败
└─ 触发沙箱重建流程

步骤 3: 沙箱重建
├─ 选择新节点 (负载最低)
├─ 新 Echo-Agentic-Engine 创建沙箱
│   └─ NAS 数据自动恢复 (共享存储)
│   └─ 启动 OpenCode Serve
├─ 更新 Redis 沙箱位置
└─ 请求正常处理
```

### 4.3 完整请求链路(简化版)

```
用户(OpenCode UI)
  ↓ HTTP 请求: POST /trans/api/chat
  ↓ Headers: X-Domain-ID, X-Sandbox-ID
Echo-Agent
  ├─ 白名单验证
  ├─ 查询 Redis(沙箱状态 + Echo-Agentic-Engine IP)
  ├─ 心跳检查(Echo-Agentic-Engine 在线?)
  └─ 路由决策
      ├─ 沙箱存活 → 转发到目标 Echo-Agentic-Engine
      └─ 沙箱不存在 → 转发到空闲 Echo-Agentic-Engine

Echo-Agentic-Engine
  ├─ 本地沙箱检查
  ├─ 不存在 → 创建沙箱 + 启动 opencode
  ├─ 更新 Redis(沙箱信息 + 本机 IP)

### 4.4 Echo-Agentic-Engine 处理流程（伪代码）

```
func handleRequest(req *http.Request) *http.Response {
    // 1. 提取请求标识
    domainID := req.Header.Get("X-Domain-ID")
    sandboxID := req.Header.Get("X-Sandbox-ID")

    // 2. 白名单验证
    if !whitelist.Contains(domainID) {
        return Response{StatusCode: 403, Body: "domain not allowed"}
    }

    // 3. 检查本地沙箱
    sandbox := localSandboxes.Get(domainID, sandboxID)

    if sandbox == nil {
        // 3.1 创建沙箱
        sandbox = createSandbox(domainID, sandboxID)
        // 3.2 启动 OpenCode Serve
        sandbox.StartOpenCode()
        // 3.3 注册到 Redis
        redis.Set("sandbox:"+domainID+":"+sandboxID, {
            "engine_ip": localIP,
            "status": "running",
            "pid": sandbox.PID,
            "port": sandbox.Port,
        })
    }

    // 4. 路径转换：去掉 /trans 前缀
    originalPath := req.URL.Path
    targetPath := strings.TrimPrefix(originalPath, "/trans")

    // 5. 构建转发请求
    forwardReq := req.Clone(context.Background())
    forwardReq.URL.Path = targetPath
    forwardReq.URL.RawPath = targetPath
    forwardReq.Header.Del("X-Domain-ID")    // 移除标识 header
    forwardReq.Header.Del("X-Sandbox-ID")

    // 6. 透传请求到沙箱内的 OpenCode Serve
    targetURL := fmt.Sprintf("http://127.0.0.1:%d%s", sandbox.Port, targetPath)
    return httpProxy.Forward(forwardReq, targetURL)
}

## 7. 设计原则

- **轻量**: 纯透传 opencode serve 接口,最小化额外逻辑
- **无状态**: 依赖 redis 和 NAS,Echo-Agentic-Engine 节点可水平扩展
- **高可用**: 多节点共享存储,支持故障恢复

## 8. 安全机制

### 8.1 白名单控制
- **配置**: 人工维护 domainID 白名单到 redis
- **更新**: Echo-Agentic-Engine 启动时抓取,后续监听 redis 变更事件
- **验证**: 每次请求校验 domainID 合法性,拒绝非白名单请求

### 8.2 沙箱隔离

**Bubblewrap 隔离机制**:
- **PID namespace**: 进程树隔离,看不到宿主进程
- **Mount namespace**: 文件系统隔离,独立挂载视图
- **IPC namespace**: 进程间通信隔离
- **UTS namespace**: 主机名隔离
- **User namespace** (可选): UID/GID 映射,非特权用户运行

**安全特性**:
- ✅ 最小权限原则: 只挂载必需目录
- ✅ 只读挂载: 系统目录使用 `--ro-bind`
- ✅ 进程隔离: `--unshare-pid` 防止进程逃逸
- ✅ 自动清理: `--die-with-parent` 防止孤儿进程
- ✅ 会话隔离: `--new-session` 防止终端劫持

**与 Docker 对比**:
| 特性 | Bubblewrap | Docker |
|------|-----------|--------|
| 攻击面 | 极小（几千行代码） | 较大（完整运行时） |
| 守护进程 | 无 | 有（dockerd） |
| 特权要求 | 非特权用户 | 需要 root/docker 组 |
| 配置复杂度 | 高（手动配置） | 低（预设模板） |
| 安全性 | 默认安全 | 依赖配置 |

### 8.3 文件系统安全

**隔离策略**:
```
沙箱内可见:
/sandbox/defaultProject/    # 读写挂载（NAS）
/usr /lib /lib64 /bin       # 只读挂载（系统）
/tmp /var                   # tmpfs（临时）
/proc /dev                  # 虚拟文件系统

沙箱内不可见:
宿主机的其他目录
其他沙箱的 NAS 目录
```

**NAS 隔离**:
- 按 domainID 隔离存储空间
- 不同 domainID 的沙箱无法互相访问
- 同一 domainID 下的不同 sandboxID 也相互隔离

### 8.4 网络安全

**网络策略（待确认）**:
- 方案 1: `--unshare-net` 完全隔离网络（如果 opencode 不需要联网）
- 方案 2: `--share-net` 共享宿主网络（如果 opencode 需要联网）
- 方案 3: 配置网络命名空间 + 白名单（高级需求）

**风险**:
- 如果共享网络，沙箱内进程可以访问外部网络
- 可能需要配置防火墙规则限制网络访问

### 9.2 故障场景与恢复策略

| 故障类型 | 检测方式 | 恢复策略 | RTO |
|---------|---------|---------|-----|
| 沙箱进程崩溃 | 进程监控 | 自动重建, NAS 恢复 | < 10s |
| Echo-Agentic-Engine 节点故障 | 心跳超时 | 流量切换其他节点 | < 30s |
| Redis 故障 | 连接超时 | 主从切换/哨兵 | < 60s |
| NAS 故障 | 挂载检测 | 多路径/冗余 | 视存储方案 |

### 9.3 故障恢复
- **沙箱故障**: 自动重建,从 NAS 恢复 workspace
- **Echo-Agentic-Engine 节点故障**: echo-agent 检测心跳,流量切换到其他节点
- **跨节点恢复**: 新节点可基于 NAS 数据恢复沙箱状态

### 9.4 无状态设计
- Echo-Agentic-Engine 节点不持久化状态,全部存储到 redis + NAS
- 支持水平扩展,节点可随时增删
