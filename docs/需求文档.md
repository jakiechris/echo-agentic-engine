# Echo-Engine-Proxy 需求文档

## 1. 整体架构

### 1.1 系统全景

```
用户界面层
├─ OpenCode UI (聊天交互界面)
└─ 文件管理 Web UI

服务网关层
└─ Echo-Agent (请求路由 + 文件服务)

沙箱管理层
└─ Echo-Engine-Proxy (本系统)

沙箱运行层
└─ Bubblewrap Sandbox → OpenCode Serve

存储层
├─ Redis (状态 + 白名单)
└─ NAS (持久化存储)
```

### 1.2 核心组件职责

| 组件 | 职责 | 边界 |
|-----|------|------|
| **OpenCode UI** | 用户聊天交互界面 | 不感知沙箱,仅通过 echo-agent 通信 |
| **Echo-Agent** | 请求路由、负载均衡、文件管理、白名单管理 | 不直接管理沙箱,只做转发 |
| **Echo-Engine-Proxy** | 沙箱生命周期管理、请求透传 | 轻量化,不处理业务逻辑 |
| **OpenCode Serve** | AI 对话服务、session 管理 | 单实例单 project,原生 API |
| **文件 Web UI** | 文件上传/下载 | 由 echo-agent 独立实现 |

## 2. Proxy 核心功能

### 2.1 请求路由
- 每个请求通过 `domainID` + `sandboxID` 唯一标识沙箱
- domainID 白名单验证(从 redis 获取,支持动态更新)
- 非法 domainID 拒绝请求

### 2.2 沙箱管理

**创建沙箱**
```bash
bwrap \n  --ro-bind /usr /usr \                      # 只读挂载系统目录
  --ro-bind /lib /lib \n  --ro-bind /lib64 /lib64 \n  --ro-bind /bin /bin \n  --bind /nas/{domainID}/{sandboxID} /sandbox \  # 读写挂载 NAS
  --tmpfs /tmp \                              # 临时文件系统
  --tmpfs /var \n  --proc /proc \                              # proc 文件系统
  --dev /dev \                                # 设备文件系统
  --unshare-pid \                             # 隔离进程命名空间
  --unshare-ipc \                             # 隔离 IPC 命名空间
  --unshare-uts \                             # 隔离主机名
  --new-session \                             # 新会话防终端劫持
  --die-with-parent \                         # 父进程退出时清理
  --setenv PATH /usr/bin:/bin \n  -- /bin/bash -c "cd /sandbox/defaultProject && opencode serve start"
```

**关键参数说明**:
- `--bind` vs `--ro-bind`: NAS 使用读写挂载，系统目录只读
- `--die-with-parent`: proxy 崩溃时自动清理沙箱
- `--new-session`: 安全隔离，防止终端劫持
- 网络: 暂不隔离（`--unshare-net` 待确认 opencode 是否需要联网）

**生命周期管理**:
- **创建**: 检查本地沙箱 → 不存在则创建 → 启动 opencode serve
- **查询**: 查询沙箱状态和位置信息
- **销毁**: 管理接口支持杀掉/启动沙箱
- **恢复**: ⚠️ **关键问题** - 需确认 session 恢复机制

### 2.3 Session 管理（关键问题）

**设计缺陷**:
- 当前无法确定恢复 project 下的哪个 session
- 需要确认 OpenCode Serve API 是否包含 sessionID 参数

**待确认项**:
1. API 请求中是否携带 sessionID？
2. 响应中是否返回 sessionID？
3. Session 创建/恢复/切换的 API 定义？
4. Session 持久化机制？

**临时方案**:
- 在 Redis 中存储: `domainID:sandboxID:sessionID` 映射
- 请求透传时从 Redis 获取 sessionID 并注入

### 2.4 状态同步
- 沙箱创建后,将信息 + 本机 IP 写入 redis
- 沙箱存活状态实时同步
- Session 信息同步（待实现）

## 3. 存储架构

### 3.1 NAS 共享存储

**挂载方式**:
```bash
# 宿主机挂载 NAS（CIFS/NFS/本地磁盘）
mount -t cifs //nas-server/share /mnt/nas -o user=username
# 或
mount -t nfs nas-server:/export/data /mnt/nas

# Bubblewrap 沙箱内访问
bwrap --bind /mnt/nas /nas ...
```

**跨节点共享**:
- 多 proxy 节点共享同一 NAS
- 支持沙箱跨节点恢复
- 文件操作独立于沙箱运行状态

### 3.2 目录结构

**沙箱根目录**:
```
/nas/{domainID}/{sandboxID}/       # 沙箱根目录
  └── defaultProject/              # opencode workspace (必须此名称)
      ├── .config/
      │   └── .opencode/
      │       └── opencode.json    # 工程级配置
      ├── .sessions/               # session 数据（待确认）
      └── [用户文件]                # workspace 文件
```

**从沙箱内视角**:
```
/sandbox/                          # 沙箱根挂载点
  └── defaultProject/              # opencode workspace
      ├── .config/.opencode/opencode.json
      └── [用户文件]
```

**OpenCode 启动**:
```bash
cd /sandbox/defaultProject
opencode serve start  # 具体命令待确认
```

### 3.3 配置文件

**opencode.json**:
- 位置: `/defaultProject/.config/.opencode/opencode.json`
- 作用: 工程级配置（API keys、模型参数等）
- 格式: 待查阅 OpenCode 官方文档

### 3.4 数据持久化

**Redis 存储**:
```
# 沙箱元数据
sandbox:{domainID}:{sandboxID} = {
  "proxy_ip": "x.x.x.x",
  "status": "running",
  "pid": 12345,
  "created_at": "2026-03-13T10:00:00Z",
  "session_id": "session-xxx"  # 待确认
}

# 白名单

# 负载信息

```

**NAS 存储**:
- Workspace 文件
- Session 数据（依赖 OpenCode 实现）
- 配置文件

**约定**:
- 单沙箱对应单 project(default),不支持多 project
- session 由 opencode API 原生管理
- 文件路径固定为 `/defaultProject/`

## 4. 系统交互流程

### 4.1 完整请求链路

```
用户(OpenCode UI)
  ↓ HTTP 请求(含 domainID + sandboxID)
Echo-Agent
  ├─ 白名单验证
  ├─ 查询 Redis(沙箱状态 + proxy IP)
  ├─ 心跳检查(proxy 在线?)
  └─ 路由决策
      ├─ 沙箱存活 → 转发到目标 proxy
      └─ 沙箱不存在 → 转发到空闲 proxy

Echo-Engine-Proxy
  ├─ 本地沙箱检查
  ├─ 不存在 → 创建沙箱 + 启动 opencode
  ├─ 更新 Redis(沙箱信息 + 本机 IP)
  └─ 透传请求到 opencode serve

OpenCode Serve
  └─ 处理 AI 对话(session 管理)
```

### 4.2 Echo-Agent 核心职责

**路由与负载均衡**
- 查询 redis 获取沙箱位置
- 心跳监控 proxy 节点健康状态
- 沙箱不存在时,选择负载最低的 proxy 节点

**白名单管理**
- 维护 domainID 白名单(写入 redis)
- 请求合法性校验

**文件服务**
- 提供 Web UI 文件上传/下载
- 操作 NAS: `/nas/domainID/sandboxID/defaultProject/`
- 独立于 proxy,不依赖沙箱是否运行

### 4.3 Proxy 处理流程
```
请求到达
  ↓
检查本地沙箱
  ├─ 存在 → 直接转发
  └─ 不存在 → 创建沙箱 + 启动 opencode + 写入 redis + 转发
```

## 6. 设计原则

- **轻量**: 纯透传 opencode serve 接口,最小化额外逻辑
- **无状态**: 依赖 redis 和 NAS,proxy 节点可水平扩展
- **高可用**: 多节点共享存储,支持故障恢复

## 7. 安全机制

### 7.1 白名单控制
- **配置**: 人工维护 domainID 白名单到 redis
- **更新**: proxy 启动时抓取,后续监听 redis 变更事件
- **验证**: 每次请求校验 domainID 合法性,拒绝非白名单请求

### 7.2 沙箱隔离

**Bubblewrap 隔离机制**:
- **PID namespace**: 进程树隔离,看不到宿主进程
- **Mount namespace**: 文件系统隔离,独立挂载视图
- **IPC namespace**: 进程间通信隔离
- **UTS namespace**: 主机名隔离
- **User namespace** (可选): UID/GID 映射,非特权用户运行

**安全特性**:
- ✅ 最小权限原则: 只挂载必需目录
- ✅ 只读挂载: 系统目录使用 `--ro-bind`
- ✅ 进程隔离: `--unshare-pid` 防止进程逃逸
- ✅ 自动清理: `--die-with-parent` 防止孤儿进程
- ✅ 会话隔离: `--new-session` 防止终端劫持

**与 Docker 对比**:
| 特性 | Bubblewrap | Docker |
|------|-----------|--------|
| 攻击面 | 极小（几千行代码） | 较大（完整运行时） |
| 守护进程 | 无 | 有（dockerd） |
| 特权要求 | 非特权用户 | 需要 root/docker 组 |
| 配置复杂度 | 高（手动配置） | 低（预设模板） |
| 安全性 | 默认安全 | 依赖配置 |

### 7.3 文件系统安全

**隔离策略**:
```
沙箱内可见:
/sandbox/defaultProject/    # 读写挂载（NAS）
/usr /lib /lib64 /bin       # 只读挂载（系统）
/tmp /var                   # tmpfs（临时）
/proc /dev                  # 虚拟文件系统

沙箱内不可见:
宿主机的其他目录
其他沙箱的 NAS 目录
```

**NAS 隔离**:
- 按 domainID 隔离存储空间
- 不同 domainID 的沙箱无法互相访问
- 同一 domainID 下的不同 sandboxID 也相互隔离

### 7.4 网络安全

**网络策略（待确认）**:
- 方案 1: `--unshare-net` 完全隔离网络（如果 opencode 不需要联网）
- 方案 2: `--share-net` 共享宿主网络（如果 opencode 需要联网）
- 方案 3: 配置网络命名空间 + 白名单（高级需求）

**风险**:
- 如果共享网络，沙箱内进程可以访问外部网络
- 可能需要配置防火墙规则限制网络访问

## 8. 高可用设计

### 8.1 故障恢复
- **沙箱故障**: 自动重建,从 NAS 恢复 workspace
- **proxy 节点故障**: echo-agent 检测心跳,流量切换到其他节点
- **跨节点恢复**: 新节点可基于 NAS 数据恢复沙箱状态

### 8.2 无状态设计
- proxy 节点不持久化状态,全部存储到 redis + NAS
- 支持水平扩展,节点可随时增删

### 10.3 系统约束

**OpenCode 约束**:
- 单沙箱单 project，不支持多 project
- Workspace 目录必须为 `/defaultProject/`
- Project 名固定为 `default`，不需要传入

**Bubblewrap 约束**:
- 需要 Linux 3.8+ 内核支持 user namespace
- 非 privileged 用户运行需要 `--unshare-user`
- 网络功能有限，需要外部工具配置复杂网络

**存储约束**:
- NAS 必须在所有 proxy 节点上挂载到相同路径
- Redis 需要高可用配置
- 文件操作独立于沙箱运行状态

### 10.4 技术风险

**Session 管理（高风险）**:
- 如果 OpenCode API 不支持 sessionID，需要设计替代方案
- 可能需要拦截/修改 API 请求来注入 session 信息

**网络隔离（中风险）**:
- 如果 OpenCode 需要联网，安全隔离会受影响
- 可能需要配置网络白名单或代理

**资源管理（中风险）**:
- Bubblewrap 默认无资源限制
- 恶意沙箱可能耗尽宿主机资源
- 需要配合 systemd/cgroups 实现资源限制
