# Echo Agentic Engine 重构更新总结

## 更新时间
2026-03-20

## 重构目标

将配置管理和沙箱监控统一到ConfigSyncTask流程，实现Engine与echo-agent的协同工作。

## 主要变更

### 1. config.json 配置更新

**变更内容**:
- `tasks.configSyncInterval`: 300秒 → 5秒
- `engine.maxSandboxes`: 50 → 100
- `sandbox.idleTimeout`: 3600秒 → 7200秒
- 删除 `tasks.healthCheckInterval`
- 删除 `tasks.idleCheckInterval`

### 2. Redis 表结构重构

#### 新增引擎信息表
- **键格式**: `engine:{engine_url}`
- **字段**: engineUrl, maxSandboxes, usedSandboxes, timestamp
- **用途**: 记录引擎心跳、容量和使用情况

#### 重构沙箱信息表
- **键格式**: `sandbox:{domainID}:{sandboxID}`
- **字段**: domainID, sandboxID, engineUrl, timestamp, lastRequestTime
- **用途**: 记录沙箱归属和活跃状态

#### 删除旧表
- 删除 `sandbox:{domainID}:{sandboxID}` 的元数据表
- 所有沙箱信息统一到新表

### 3. ConfigSyncTask 重写

**原职责**: 从Redis同步配置到本地

**新职责**:
1. 每5秒向Redis写入引擎信息表
2. 检测并清理不健康沙箱
3. 同步沙箱信息表(多退少补)

**实现细节**:
- 自动获取引擎IP和URL
- 并发健康检查所有沙箱
- 不健康沙箱直接杀死(不重建)
- 对比Redis和内存,同步沙箱记录

### 4. 管理接口重构

#### 3.2.2 列出所有信息 (原列出所有沙箱)
- **新功能**: 同时返回引擎信息表和沙箱信息表
- **数据源**: 完全从Redis读取
- **跨引擎**: 可以看到所有引擎的信息

#### 3.2.3 清理空闲沙箱 (原查询单个沙箱)
- **新功能**: 清理空闲超过指定时长的沙箱
- **请求参数**: `{"idleMinutes": 5}`
- **清理范围**: 只清理本引擎负责的沙箱
- **判断依据**: lastRequestTime与当前时间差值

#### 3.2.4 创建沙箱 (更新)
- **新增**: 创建后写入Redis沙箱信息表

#### 3.2.5 销毁沙箱 (更新)
- **新增**: 销毁后从Redis删除沙箱信息

### 5. 删除的模块

#### 删除文件
- `/src/background_tasks/health_check_task.py`
- `/src/background_tasks/idle_cleanup_task.py`

#### 删除功能
- HealthCheckTask: 健康检查巡检任务
- IdleCleanupTask: 空闲沙箱清理任务

#### 删除原因
- ConfigSyncTask已包含健康检查功能
- 空闲清理策略由echo-agent控制

## 架构优势

### 原架构问题
1. Engine自行管理空闲清理,无法与echo-agent协同
2. 健康检查和空闲清理分散在两个任务,职责不清
3. Redis表结构不支持多引擎实例
4. 配置同步间隔过长(300秒),响应慢

### 新架构优势
1. **统一职责**: ConfigSyncTask统一管理状态同步和健康检查
2. **协同工作**: echo-agent通过Redis表和API接口控制清理策略
3. **多实例支持**: 引擎信息表支持多Engine实例
4. **快速响应**: 5秒同步间隔,快速发现问题
5. **状态一致性**: 内存和Redis保持一致,支持跨节点恢复

## 迁移指南

### 配置文件迁移

更新 `config.json`:
```json
{
  "engine": {
    "maxSandboxes": 100
  },
  "sandbox": {
    "idleTimeout": 7200
  },
  "tasks": {
    "configSyncInterval": 5
  }
}
```

删除旧配置项:
- `tasks.healthCheckInterval`
- `tasks.idleCheckInterval`

### Redis 数据迁移

**无需手动迁移**:
- ConfigSyncTask会自动重建Redis表
- 旧数据会在首次同步时被清理

### API 调用迁移

**3.2.2 列出沙箱**:
```bash
# 旧接口
curl -X POST http://localhost:8000/admin/sandbox/list -d '{}'

# 新接口(相同)
curl -X POST http://localhost:8000/admin/sandbox/list -d '{}'

# 响应格式变化
# 旧: {"sandboxes": [...]}
# 新: {"engines": [...], "sandboxes": [...]}
```

**3.2.3 查询沙箱 → 清理空闲沙箱**:
```bash
# 旧接口(已废弃)
curl -X POST http://localhost:8000/admin/sandbox/get -d '{"domainID":"test","sandboxID":"test"}'

# 新接口
curl -X POST http://localhost:8000/admin/sandbox/get -d '{"idleMinutes":5}'
```

## 兼容性说明

### 向后兼容
- 所有OpenCode API代理接口完全兼容
- 创建/销毁沙箱接口完全兼容

### 不兼容变更
- 3.2.3接口功能完全改变
- Redis表结构不兼容(需要重建)
- 配置文件部分字段变更

### 迁移建议
1. 停止旧版本Engine
2. 更新config.json
3. 清空Redis中的旧数据(可选)
4. 启动新版本Engine
5. ConfigSyncTask会自动建立新的Redis表

## 性能影响

### 正面影响
- 5秒心跳,故障发现更快
- 引擎信息集中管理,调度更高效
- 减少后台任务数量,降低系统负载

### 负面影响
- 5秒一次Redis写入,频率提高
- 健康检查频率从30秒提高到5秒

### 优化建议
- 如果沙箱数量很多,可适当调整configSyncInterval
- 健康检查使用线程池并发执行,性能影响可控
